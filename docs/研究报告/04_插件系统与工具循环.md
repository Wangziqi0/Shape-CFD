# 04 — 插件系统与工具循环

> Agent 4 研究报告 | 研究范围：`Plugin.js`、`toolCallParser.js`、`toolExecutor.js`、`streamHandler.js`、`nonStreamHandler.js`

---

## 一、设计理念

VCPToolBox 的插件系统本质上是一个 **文本标记驱动的 Tool-Use 框架**，与 OpenAI Function Calling 的原生 JSON Schema 方式截然不同。其核心思路是：

1. **绕过 API 限制**：不依赖 LLM 厂商的原生 tool_call 能力，而是在系统提示词中教会 AI 使用自定义文本标记来"调用工具"，从而兼容任何 Chat Completion API。
2. **代理中间件**：VCPToolBox 作为客户端与 LLM API 之间的透明代理，拦截 AI 响应、解析标记、执行插件、将结果注入消息历史后再次调用 LLM，形成闭环。
3. **插件即进程/模块**：插件可以是独立子进程（stdio 通信）、常驻内存的 Node.js 模块（direct 协议）、或远端分布式节点。

## 二、核心数据结构

### 2.1 plugin-manifest.json（插件清单）

每个插件必须包含一个 `plugin-manifest.json`，定义其元数据和能力：

```json
{
  "name": "GoogleSearch",          // 唯一标识符，也是工具调用名
  "pluginType": "synchronous",     // 类型：synchronous | asynchronous | static | messagePreprocessor | service | hybridservice
  "entryPoint": {
    "type": "nodejs",
    "command": "node search.js"    // stdio 类插件的启动命令
  },
  "communication": {
    "protocol": "stdio",           // 通信协议：stdio | direct
    "timeout": 15000
  },
  "configSchema": { ... },         // 配置项声明，支持类型校验（string/integer/boolean）
  "capabilities": {
    "invocationCommands": [...]    // 工具调用描述，注入到系统提示词中
  }
}
```

### 2.2 PluginManager 核心状态

```
plugins: Map<name, manifest>              // 本地 + 分布式插件统一管理
messagePreprocessors: Map<name, module>    // 消息预处理器（direct 协议的常驻模块）
serviceModules: Map<name, {manifest, module}> // 服务型插件（注册 Express 路由）
staticPlaceholderValues: Map<key, {value, serverId}> // 系统提示词占位符
scheduledJobs: Map<name, job>              // cron 定时任务
preprocessorOrder: string[]                // 预处理器执行顺序
```

### 2.3 工具调用标记格式

```
<<<[TOOL_REQUEST]>>>
tool_name:「始」GoogleSearch「末」,
query:「始」如何学习编程？「末」,
archery:「始」true「末」
<<<[END_TOOL_REQUEST]>>>
```

- 外层：`<<<[TOOL_REQUEST]>>>` / `<<<[END_TOOL_REQUEST]>>>` 包围
- 内层：**中日混合标记** `「始」...「末」` 作为参数值定界符（利用了 LLM 对中文标点的高识别率）
- 特殊参数：`archery`（异步fire-and-forget）、`ink`（标记历史留痕）

## 三、算法流程

### 3.1 插件生命周期

```
启动 → loadPlugins() → 遍历 Plugin/ 目录
  → 读取 plugin-manifest.json + config.env
  → 按类型分类注册到 plugins/messagePreprocessors/serviceModules
  → 读取 preprocessor_order.json 确定预处理器顺序
  → 依次调用 module.initialize(config, dependencies) 初始化
  → 特殊依赖注入（RAGDiaryPlugin ← VectorDBManager, LightMemo ← RAGDiaryPlugin）
  → buildVCPDescription() 构建工具描述 → 注入系统提示词

运行时：chokidar 监视 manifest 文件变化 → 500ms 防抖 → hotReloadPluginsAndOrder()
关闭：shutdownAllPlugins() → 逐一调用 module.shutdown()
```

### 3.2 VCP Tool Loop（核心循环）

这是整个系统最关键的算法，流式和非流式共用同一逻辑模型：

```
AI 首次响应 → 收集完整输出内容
      ↓
while (depth < maxRecursion) {
  1. ToolCallParser.parse(content) → 提取工具调用列表
  2. 若无工具调用 → 终止循环，返回给客户端
  3. ToolCallParser.separate() → 分离 normal 调用 和 archery 异步调用
  4. 并行执行 archery 调用（fire-and-forget，仅收集错误）
  5. 并行执行 normal 调用 → toolExecutor.executeAll()
  6. 合并结果 → 包装为 user 角色消息（<!-- VCP_TOOL_PAYLOAD --> 标记）
  7. 可选：RAGMemoRefresh 刷新记忆块
  8. 将 [assistant + user(结果)] 追加到消息历史
  9. 再次调用 LLM API → 获取新响应
  10. depth++
}
```

**关键设计点**：
- 工具结果以 `user` 角色注入（而非 `tool` 角色），保证任何 LLM 都能处理
- `<!-- VCP_TOOL_PAYLOAD -->` HTML 注释标记，让 AI 能区分工具结果和用户消息
- 支持图片类型结果（`image_url`），自动切换为多模态 content 格式
- Base64 图片在 RAG 序列化时自动替换为 `[Omitted]`，避免 token 爆炸

### 3.3 流式 vs 非流式的差异

| 维度 | StreamHandler | NonStreamHandler |
|------|--------------|-----------------|
| 转发策略 | 直通转发 SSE chunks + 后台解析内容 | 等待完整响应后统一处理 |
| 保活机制 | 5 秒 SSE 心跳注释 `: vcp-keepalive` | 无 |
| VCP 信息展示 | 实时流式注入 `ROLE_DIVIDE_USER` 块 | 拼接到 `conversationHistoryForClient` |
| 最终输出 | 逐 chunk 转发 + `data: [DONE]` | 单次 `res.send()` 完整 JSON |
| 终止信号 | `finish_reason: 'stop'` / `'length'` | 同上，嵌入 JSON 响应体 |

### 3.4 Archery 模式（异步 Fire-and-Forget）

Archery 是 VCP 的创新设计：AI 可以标记某些工具调用为"射后不管"，即：
- 执行工具但**不等待结果**回传给 AI（除非出错）
- 适用于写入操作（写日记、存记忆），不需要 AI 看到执行结果
- 错误时才将结果追加到消息历史，让 AI 感知问题

## 五、可借鉴点

### 1. 文本标记驱动的工具调用（核心价值）
legal-assistant 可借鉴这种**不依赖原生 Function Calling 的工具调用方式**。好处是可以在系统提示词中灵活定义工具，不受模型 API 限制。但需权衡：Token 消耗更高，且标记可能被模型写错。

### 2. 插件 Manifest 声明式注册
每个能力模块通过 `plugin-manifest.json` 自描述，PluginManager 只需遍历目录即可自动发现。这种设计可直接用于 legal-assistant 的模块化扩展。

### 3. 预处理器链的有序执行
`preprocessor_order.json` + 按序初始化，保证了模块间依赖关系（如 RAGDiaryPlugin 必须在 LightMemo 之前初始化）。legal-assistant 的 RAG 管线也需要类似的有序处理。

### 4. Archery 异步调用模式
区分"需要结果的调用"和"写入型调用"，是一种节省 LLM 轮次和 Token 的巧妙优化。legal-assistant 的案例存储、日志记录等操作可采用类似策略。

### 5. 流式保活心跳
SSE 流中每 5 秒发送 `: vcp-keepalive` 注释，防止长时间工具执行时浏览器断连。这是生产环境中非常实用的工程技巧。

### 6. 热重载与文件监视
`chokidar` 监视 manifest 变化 + 500ms 防抖 + 常驻内存插件跳过重载 = 稳定的开发体验。legal-assistant 的配置热更新可参考此模式。

### 7. 分布式插件透传
本地/分布式插件使用同一个 `plugins` Map 管理，调用时按 `isDistributed` 标志自动路由。这为 legal-assistant 未来的多节点部署提供了参考。

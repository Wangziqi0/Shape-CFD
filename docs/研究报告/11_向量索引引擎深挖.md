## 一、Vexus-Lite：唯一向量引擎（非双引擎）

### 1.1 架构定位

Vexus-Lite 是一个**自研的 NAPI-RS 桥接层**，底层封装了 Rust 生态的 `usearch` 库。**不存在 hnswlib-node 回退**——Vexus-Lite 是唯一引擎，加载失败则 `process.exit(1)` 直接退出。

```rust
// 核心结构
pub struct VexusIndex {
    index: Arc<RwLock<Index>>,  // usearch::Index
    dimensions: u32,
}
```

### 1.2 HNSW 索引参数（硬编码）

| 参数 | 值 | 含义 |
|------|-----|------|
| `metric` | `L2sq` | 平方欧氏距离（对归一化向量等价余弦） |
| `quantization` | `F32` | 32位浮点精度，无量化损失 |
| `connectivity` (M) | **16** | 每节点最大连接数 |
| `expansion_add` (efConstruction) | **128** | 构建时搜索宽度 |
| `expansion_search` (efSearch) | **64** | 查询时搜索宽度 |
| `multi` | `false` | 不允许同一 key 多次插入 |

> **设计取舍**：M=16 是平衡精度与内存的经典选择。efSearch=64 偏向低延迟而非极致召回率。

### 1.3 得分转换公式

```rust
// L2sq 距离 → 相似度分数
score = 1.0 - dist  // dist ∈ [0, +∞)，score ≤ 1.0
```

> ⚠️ 这是一个**近似转换**，仅对归一化向量有效（dist ≈ 2(1-cosine)）。KBM 中向量在融合后会重新归一化，保证此公式合理。

### 1.4 自动扩容机制

```rust
if index.size() + 1 >= index.capacity() {
    let new_cap = (index.capacity() as f64 * 1.5) as usize;
    let _ = index.reserve(new_cap);
}
```

- 初始容量：Tag 索引和日记索引均为 **50000**
- 扩容策略：当前容量 × 1.5（几何增长）
- 扩容时机：每次 add 前检查

### 1.5 原子保存

```rust
// 先写 .tmp 再 rename，防止写入中断导致索引损坏
let temp_path = format!("{}.tmp", index_path);
index.save(&temp_path)?;
std::fs::rename(&temp_path, &index_path)?;
```

## 二、索引生命周期管理

### 2.1 索引类型与存储

| 索引 | 文件名 | 容量 | 驱逐策略 |
|------|--------|------|----------|
| 全局 Tag 索引 | `index_global_tags.usearch` | 50000 | 永驻内存 |
| 日记本索引 | `index_diary_{md5(name)}.usearch` | 50000 | TTL 驱逐 |

### 2.2 日记本索引懒加载完整流程

```
用户查询某日记本
  │
  ├─→ 检查 diaryIndices Map（内存缓存）
  │     ├─ 命中 → 刷新 lastUsed 时间戳 → 返回索引
  │     └─ 未命中 ↓
  │
  ├─→ 检查 .usearch 文件是否存在
  │     ├─ 存在 → VexusIndex.load() 加载到内存
  │     └─ 不存在 → new VexusIndex() 空索引
  │
  ├─→ 加载失败 → new VexusIndex() + _recoverIndexFromDB()
  │     └─ Rust 端从 SQLite 读取 vector BLOB 重建索引
  │
  └─→ 存入 diaryIndices Map → 返回索引
```

### 2.3 TTL 驱逐机制

```javascript
// 配置
indexIdleTTL: 2 * 60 * 60 * 1000,      // 2小时空闲超时
indexIdleSweepInterval: 10 * 60 * 1000,  // 每10分钟扫描一次

// 驱逐流程
_evictIdleIndices() {
    // 1. 检查每个索引的最后使用时间
    // 2. 超过 TTL → 先 _saveIndexToDisk() 保存
    // 3. diaryIndices.delete() + diaryIndexLastUsed.delete()
    // 4. timer.unref() 允许 Node 自然退出
}
```

### 2.4 索引保存调度

采用**延迟合并保存**策略，避免频繁写盘：

- 日记索引：变更后延迟 **120 秒** 保存
- Tag 索引：变更后延迟 **300 秒** 保存
- shutdown 时立即保存所有待保存索引

## 三、SQLite BLOB 向量序列化

### 3.1 存储格式

```javascript
// 写入：Float32Array → Buffer（原始字节）
const vecBuf = Buffer.from(new Float32Array(vec).buffer);
// ↑ 直接拷贝 Float32Array 的 ArrayBuffer，每个浮点 4 字节

// 读取：Buffer → Float32Array（零拷贝视图）
const vec = new Float32Array(row.vector.buffer, row.vector.byteOffset, dimension);
// ↑ 使用 byteOffset 确保 Buffer 视图正确（better-sqlite3 返回的 Buffer 可能有偏移）
```

以 3072 维向量为例：每个向量 = `3072 × 4 = 12288 字节`。

### 3.2 维度校验

Rust 端通过字节长度验证维度：

```rust
let expected_byte_len = dimensions * sizeof::<f32>();  // dim * 4
if vector_bytes.len() == expected_byte_len { /* OK */ }
else { skipped_dim_mismatch += 1; }
```

## 四、`recoverFromSqlite()` 重建算法

### 4.1 Rust 异步任务实现

```rust
impl Task for RecoverTask {
    fn compute(&mut self) -> Result<u32> {
        // 1. 打开 SQLite 连接
        let conn = Connection::open(&self.db_path)?;
        
        // 2. 根据 table_type 构建 SQL
        //    - "tags"   → SELECT id, vector FROM tags
        //    - "chunks" → SELECT c.id, c.vector FROM chunks c JOIN files f ...
        //                 WHERE f.diary_name = ?1
        
        // 3. 获取索引写锁
        let index = self.index.write()?;
        
        // 4. 遍历结果行
        for (id, vector_bytes) in rows {
            // 维度校验
            if vector_bytes.len() == expected_byte_len {
                // 自动扩容
                if index.size() + 1 >= index.capacity() {
                    index.reserve((capacity * 1.5) as usize);
                }
                // 添加向量
                index.add(id as u64, vec_slice);
                count += 1;
            }
        }
        Ok(count)
    }
}
```

**关键设计**：使用 NAPI-RS 的 `AsyncTask` 在独立线程中执行，不阻塞 Node.js 事件循环。

## 五、搜索得分校准与完整检索链路

### 5.1 单日记本检索 `_searchSpecificIndex()`

```
输入: (diaryName, queryVector, k, tagBoost, coreTags)
  │
  ├─→ _getOrLoadDiaryIndex(diaryName)  // 懒加载
  │
  ├─→ tagBoost > 0 ?
  │     ├─ Yes → _applyTagBoostV6() → 增强后的搜索向量
  │     └─ No  → 使用原始向量
  │
  ├─→ Float32Array → Buffer 转换（byteOffset 安全处理）
  │
  ├─→ 维度校验
  │
  ├─→ idx.search(searchBuffer, k)  // Vexus Rust 搜索
  │     └─ 返回 [{id, score}] （score = 1.0 - L2sq_dist）
  │
  └─→ Hydrate：SQLite 查询 chunks.content + files.path
       └─ 附加 tagInfo（matchedTags, boostFactor, coreTagsMatched）
```

### 5.2 全局搜索 `_searchAllIndices()`

```
TagMemo 增强（如果启用）→ 计算一次增强向量
  │
  ├─→ 查询所有日记本名称（SELECT DISTINCT diary_name FROM files）
  │
  ├─→ Promise.all 并行搜索每个日记本索引
  │     └─ 每个索引返回 top-k 结果
  │
  ├─→ 合并所有结果 → 按 score 降序排序
  │
  └─→ 取全局 top-k → Hydrate
```

> **设计亮点**：TagMemo 增强向量只计算一次，全局共享。

## 六、V7 内生残差 Rust 预计算

### 6.1 算法流程

```
对每个 Tag T:
  1. 从 file_tags 表构建邻居列表（共现的其他 Tag）
  2. 收集邻居的向量矩阵 M (n×dim, n≤100)
  3. SVD 分解: M = U·S·Vᵀ，截断取前 k 个主成分 (k≤8)
  4. 计算 T 在邻居子空间的投影: proj = Σᵢ (T·vᵢ)·vᵢ
  5. 残差能量: residual = ||T - proj||₂
  6. 归一化到 [0.5, 2.0]: normalized = 0.5 + 1.5 × (r - r_min) / (r_max - r_min)
```

### 6.2 关键参数

| 参数 | 默认值 | 含义 |
|------|--------|------|
| `max_svd_rank` | 8 | 最大保留主成分数 |
| `min_neighbors` | 3 | 最少邻居数（少于则跳过） |
| `max_neighbors` | 100 | 最大邻居数（防 SVD 爆炸） |

### 6.3 残差语义

- **残差高 → 接近 2.0**：该 Tag 独特，不能被邻居线性表达 → 脉冲扩散时**增益更大**
- **残差低 → 接近 0.5**：该 Tag 冗余，可被邻居近似 → 脉冲扩散时**增益较小**

在 V6 LIF 脉冲传播中的使用：

```javascript
const nodeResidual = this.tagIntrinsicResiduals?.get(nodeId) ?? 1.0;
const injectedCurrent = energy * coocWeight * DECAY_FACTOR * nodeResidual;
```

## 七、有向共现矩阵构建算法

### 7.1 序位势能 PHI

```javascript
// PHI_MAX = 0.9, PHI_MIN = 0.5
// position: Tag 在文件中的排列序号（1-indexed）
phi(pos) = PHI_MAX - (PHI_MAX - PHI_MIN) * (pos - 1) / (n - 1)
// 第1个 Tag: phi = 0.9（最高势能）
// 最后一个 Tag: phi = 0.5
// 边权重: weight = phi(source) × phi(target)
```

### 7.2 有向边构建

```sql
SELECT ft1.tag_id as source, ft2.tag_id as target, ft1.position, ft2.position
FROM file_tags ft1 JOIN file_tags ft2
ON ft1.file_id = ft2.file_id AND ft1.position < ft2.position
-- 只沿 position 递增方向建边（有向）
```

### 7.3 旧数据兼容

对 `position = 0` 的旧数据，退化为**无向等权重**边（`LEGACY_PHI = 0.7`）。

### 7.4 矩阵重建调度（V7.7 混合防抖）

```
文件更新 → 记录 actualTagChanges
  │
  ├─→ 累计变动 < 阈值（max(10, min(200, totalTags×1%))）→ 不触发
  │
  └─→ 累计变动 ≥ 阈值 → 启动 5 分钟滑动窗口防抖
       └─ 5 分钟静默期后执行：
          1. _buildDirectedCooccurrenceMatrix()
          2. recomputeIntrinsicResiduals()（Rust 异步计算）
```

## 八、EmbeddingUtils 向量化管线

### 8.1 并发批处理架构

```
输入 N 个文本
  │
  ├─→ tiktoken (cl100k_base) 计算每个文本的 token 数
  │     └─ 超过 safeMaxTokens (embeddingMaxToken×0.85) → 跳过，位置填 null
  │
  ├─→ 批次打包：受双限制
  │     ├─ token 限制：单批总 token < safeMaxTokens
  │     └─ 条目限制：单批 ≤ 100 条
  │
  ├─→ 并发 Worker 模式
  │     ├─ 启动 N 个 Worker（默认 5，可配置 TAG_VECTORIZE_CONCURRENCY）
  │     ├─ 每个 Worker 通过 cursor++ 抢任务（类协程调度）
  │     └─ 单 batch 失败不影响其他 batch
  │
  └─→ 按 originalIndex 回填结果
       └─ 保证 output.length === input.length（失败位置为 null）
```

### 8.2 重试策略

- 3 次重试，指数退避（`baseDelay × 2^attempt`）
- 429 限流：等待 `5000ms × attempt`

## 十一、可借鉴点

### 11.1 Rust 高性能桥接 ⭐⭐⭐
通过 NAPI-RS 将计算密集操作（SVD、索引恢复）卸载到 Rust 线程。legal-assistant 可借鉴此模式，用 Rust/C++ 加速向量操作。

### 11.2 懒加载 + TTL 驱逐 ⭐⭐⭐
多数据库场景下的内存管理最佳实践。legal-assistant 若按法律领域分索引（劳动法、合同法等），可直接复用此策略。

### 11.3 热调控参数文件 ⭐⭐
`rag_params.json` + chokidar 监听实现无重启调参。legal-assistant 的检索参数（召回数、阈值等）也应支持热更新。

### 11.4 向量化管线的容错设计 ⭐⭐⭐
EmbeddingUtils 的"失败填 null + 长度对齐"保证是极重要的工程细节。防止单个超长文本拖垮整个批处理。

### 11.5 原子写入模式 ⭐
`.tmp + rename` 防止索引文件损坏，简单但关键的可靠性保障。

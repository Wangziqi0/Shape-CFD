# 03 · 请求处理与消息管线 — VCPToolBox 研究报告

> **Agent 3** | 研究范围：`chatCompletionHandler.js`、`messageProcessor.js`、`roleDivider.js`  
> 总代码量 ~74KB（1659 行）

---

## 一、设计理念

VCPToolBox 的请求处理采用**"管线（Pipeline）模式"**——一个 OpenAI 兼容的 `/v1/chat/completions` 请求从进入到发出，会经历 **7 个可配置的处理阶段**，每个阶段只做一件事，顺序执行且阶段间通过 `messages` 数组传递状态。

核心设计哲学：**"预设是死的，管线是活的"**。用户写的系统提示词（预设）中充满了 `{{变量}}` 占位符，这些占位符在运行时由管线动态展开为实际内容——时间、Agent 人格、插件描述、RAG 检索结果、动态折叠内容等。这使得一份静态预设能根据上下文产生截然不同的运行时提示词。

## 二、核心数据结构

### 2.1 消息对象（Message Object）

标准 OpenAI 格式 `{ role, content }`，其中 `content` 可以是：
- **字符串**：纯文本消息
- **数组**：多模态消息（含 text/image_url 等 part）

### 2.2 角色分割标签（Role Divider Tags）

```
<<<[ROLE_DIVIDE_SYSTEM]>>>  ... <<<[END_ROLE_DIVIDE_SYSTEM]>>>
<<<[ROLE_DIVIDE_ASSISTANT]>>> ... <<<[END_ROLE_DIVIDE_ASSISTANT]>>>
<<<[ROLE_DIVIDE_USER]>>>    ... <<<[END_ROLE_DIVIDE_USER]>>>
```

一条消息中嵌入这些标签后，RoleDivider 会将其**拆分为多条不同角色的消息**。这允许用户在一个预设文本框中定义多角色对话结构。

### 2.3 变量占位符体系

| 占位符类型 | 示例 | 作用域 |
|-----------|------|--------|
| Agent 变量 | `{{AgentName}}` / `{{agent:AgentName}}` | 全角色 |
| 时间变量 | `{{Date}}` `{{Time}}` `{{Today}}` `{{Festival}}` | system |
| 环境变量 | `{{TarXxx}}` `{{VarXxx}}` | system |
| 模型条件注入 | `{{SarPrompt1}}` + `SarModel1` | system/特定user |
| 插件占位符 | `{{VCPAllTools}}` `{{插件名Desc}}` | system |
| 表情包变量 | `{{XXX表情包}}` | system |
| 异步结果 | `{{VCP_ASYNC_RESULT::插件名::请求ID}}` | 全角色 |
| 动态折叠 | `vcp_dynamic_fold` 协议对象 | system |

### 2.4 RAG 区块标记

```html
<!-- VCP_RAG_BLOCK_START {json元数据} -->
  ...检索内容...
<!-- VCP_RAG_BLOCK_END -->
```

在 VCP 工具循环中，历史消息里的 RAG 区块会被**动态刷新**——基于新上下文重新检索，保证多轮对话中 RAG 内容的新鲜度。

## 三、算法流程

### 3.1 完整请求处理管线（7 阶段）

```
请求进入
  │
  ▼
① 上下文修剪（contextTokenLimit → contextManager.pruneMessages）
  │
  ▼
② 模型重定向 + 国产模型思维链控制
  │
  ▼
③ RoleDivider 角色分割（Initial Stage）
  │    → 将预设中的角色标签拆分为独立消息
  │
  ▼
④ VCPTavern 优先预处理（注入预设内容）
  │
  ▼
⑤ 统一变量替换（resolveAllVariables）
  │    → Agent 解析（递归展开，防循环依赖）
  │    → 优先级变量（表情包等）
  │    → 其他变量（时间/环境/Detector/插件/动态折叠/异步结果）
  │
  ▼
⑥ 消息预处理器链（媒体处理器 → RAG 插件 → 其他插件）
  │
  ▼
⑦ fetchWithRetry 发送请求 → 流式/非流式分发处理
```

### 3.2 Agent 变量递归解析算法

`resolveAllVariables()` 是变量系统的核心，其算法要点：

1. **正则扫描**所有 `{{xxx}}` 占位符
2. **通过 agentManager 判断**该占位符是否指向一个 Agent 人格文件
3. **读取 Agent 提示词内容**（可能又包含 `{{其他Agent}}`）
4. **递归展开**，使用 `processingStack`（Set）检测循环依赖
5. 所有 Agent 展开完毕后，再处理剩余的非 Agent 变量

```javascript
// 循环依赖检测
if (processingStack.has(alias)) {
    // 报错并替换为错误提示
    return `[Error: Circular agent reference for '${alias}']`;
}
processingStack.add(alias);
const resolved = await resolveAllVariables(content, ...);
processingStack.delete(alias);
```

**设计亮点**：Agent 变量优先于其他变量处理，确保 Agent 文件中引用的 `{{Time}}` 等变量也能被正确展开。

### 3.3 动态上下文折叠算法（Dynamic Fold）

这是 messageProcessor 中最独特的算法——**基于语义相似度自适应调整注入内容的详细程度**：

1. 插件注册占位符时提供一个 `vcp_dynamic_fold` 对象，包含多个 `fold_blocks`（每个有 `threshold` 和 `content`）
2. 处理时，提取**最近的用户和 AI 消息**作为上下文
3. 调用 RAG 插件获取上下文的**向量嵌入**（命中缓存）
4. 获取**插件描述的向量嵌入**（SQLite 持久化缓存 or 内存缓存）
5. 计算两个向量的**余弦相似度**
6. 按阈值降序匹配，相似度越高 → 展开越详细的内容

```
相似度 ≥ 0.7 → 展开完整的工具使用指南
相似度 ≥ 0.5 → 展开简化版本
相似度 < 0.5 → 仅展开一句话概述
```

**设计亮点**：动态折叠让系统提示词的长度随对话主题自适应变化——当用户聊到某个插件相关话题时自动展开该插件的详细说明，否则折叠为极短描述，节省 token。

### 3.4 RoleDivider 角色分割算法

RoleDivider 通过**基于游标的单遍扫描**将一条消息拆分为多条：

1. **保护块识别**：先扫描所有 `TOOL_REQUEST` 和 `DailyNote` 区块，标记为受保护（不参与分割）
2. **预扫描**：收集所有有效的 START/END 标签位置（排除保护块内的）
3. **游标遍历**：从头到尾扫描文本
   - 遇到 START 标签 → 将之前的缓冲区作为原角色消息推入结果，标签内容作为新角色消息
   - 遇到 END 标签（无配对 START）→ **健壮性处理**：将之前的缓冲区作为标签角色消息
4. **忽略列表**：通过 `normalizeForIgnore`（去除空白/换行/反斜杠后比较）跳过特定内容
5. **多模态支持**：对数组内容递归处理每个 text 类型的 part

**健壮性设计**：
- 无 END 标签的 START → 检查全文是否有任何 END 标签，若没有则将后续文本全部作为新角色
- 无 START 标签的 END → 检查全文是否有任何 START 标签，若没有则将之前文本作为标签角色
- 开关控制 `switches` / `scanSwitches`：可精细控制哪些角色参与分割、哪些消息角色被扫描

### 3.5 fetchWithRetry 重试机制

```
特性：
├── 指数退避重试（delay * attemptNumber）
├── 可重试状态码：500、503、429
├── 连接超时安全网（默认 120s）
├── 超时中止 vs 外部中止（用户取消）的区分
├── 外部中止信号转发（通过 AbortController 链式传递）
└── onRetry 回调（用于在流式模式下提前发送 200 响应头）
```

### 3.6 RAG 区块动态刷新

`_refreshRagBlocksIfNeeded()` 在 VCP 工具循环的每一轮中被调用：

1. 正则匹配历史消息中所有 `VCP_RAG_BLOCK` 标记
2. 解析元数据 JSON（含 dbName 等信息）
3. **回溯查找最近的真实用户消息**（排除工具载荷和系统提示）
4. 调用 RAG 插件的 `refreshRagBlock()` 接口重新检索
5. 使用 `replace(str, () => newBlock)` 替换，**防止 `$` 符号导致的正则替换 bug**

### 3.7 Detector / SuperDetector 系统

- **Detector**：仅对 `system` 角色消息生效，将预设中的特定文本替换为另一段文本
- **SuperDetector**：对**所有角色**消息生效，优先级更高

用途：针对不同模型的系统提示词微调（如某些模型不理解特定指令格式时进行转换）。

## 四、关键代码片段

### 4.1 动态折叠的余弦相似度计算

```javascript
let dotProduct = 0, normA = 0, normB = 0;
const len = Math.min(descVector.length, userVector.length);
for (let i = 0; i < len; i++) {
    dotProduct += descVector[i] * userVector[i];
    normA += descVector[i] * descVector[i];
    normB += userVector[i] * userVector[i];
}
const sim = (normA === 0 || normB === 0)
    ? 0
    : dotProduct / (Math.sqrt(normA) * Math.sqrt(normB));
```

## 五、可借鉴点（面向 legal-assistant）

### 5.1 管线式消息预处理架构 ⭐⭐⭐
将消息预处理拆分为**有序的、可插拔的阶段**。每个阶段（变量替换、RAG 注入、PII 脱敏等）作为独立模块，通过顺序执行 + messages 数组传递实现松耦合。legal-assistant 可据此设计统一的预处理管线。

### 5.2 动态上下文折叠 ⭐⭐⭐
**最有价值的借鉴点**。legal-assistant 的系统提示词中包含大量法律领域知识（劳动法、合同法等），可以用动态折叠技术——当用户问劳动法问题时自动展开劳动法相关指引，折叠其他领域，显著节省 token。

### 5.3 角色分割机制 ⭐⭐
RoleDivider 允许在单条消息内定义多角色内容。在 legal-assistant 中可用于：将法律条文引用（system 角色）和对话回复（assistant 角色）在模板中灵活组织。

### 5.4 RAG 区块动态刷新 ⭐⭐⭐
多轮对话中 RAG 检索结果的"保鲜"机制。legal-assistant 在多轮法律咨询中，早期检索的法条可能与后续话题偏离，该机制可保证每轮对话中引用的法条始终与当前讨论相关。

### 5.5 变量递归解析 + 循环检测 ⭐⭐
Agent 提示词的递归展开和循环依赖检测是成熟的模板系统设计。legal-assistant 若引入"法律顾问角色模板"系统，可复用此设计。

### 5.6 fetchWithRetry 超时分层策略 ⭐
区分"连接超时"和"用户取消"的中止处理，以及指数退避重试，是稳健的 API 调用实践。

### 5.7 Detector 文本转换规则 ⭐
可用于 legal-assistant 中针对不同 LLM 模型的提示词格式适配（不同模型对法律推理的指令格式偏好不同）。

## 二、核心算法：LightMemo 三阶段混合检索

LightMemo 是最核心的模块（35KB），实现了一套完整的 **BM25 + 向量 + Rerank** 三阶段检索管线。

### 2.1 第一阶段：BM25 关键词初筛

```
查询文本 → jieba分词 → 语义组扩展 → BM25评分 → 取TopK×5
```

**BM25Ranker 类**实现经典 BM25 公式：
- `k1=1.5`（词频饱和参数）、`b=0.75`（文档长度惩罚参数）
- IDF 采用 `log((N - df + 0.5) / (df + 0.5) + 1)` 平滑公式
- 分词使用 `@node-rs/jieba` Rust 绑定，性能优于纯 JS 实现
- **停词表**：28 个高频中文虚词手动定义

**语义组扩展（Query Expansion）**：  
查询词命中语义组后，将组内所有同义词加入查询。组数据来自 `semantic_groups.json`，由 SemanticGroupEditor 维护。这是一种**受控同义词扩展**策略，不依赖 LLM，延迟低。

**兜底机制**：若 BM25 匹配数 < k，则补充未匹配的候选进入下一阶段，确保向量检索有足够候选。

### 2.2 第二阶段：向量精排

```
查询向量 → TagMemo向量增强(可选) → 余弦相似度评分 → 混合评分
```

- **TagMemo V6 向量增强**：调用 `vectorDBManager.applyTagBoost()` 对查询向量施加标签方向偏移，支持 `core_tags`（核心标签额外放大 `core_boost_factor=1.33`）
- **混合评分公式**：
  - 有 BM25 分时：`hybridScore = 0.6 × normalize(BM25) + 0.4 × vectorScore`
  - 无 BM25 分时：`hybridScore = vectorScore`（**动态权重**，避免关键词缺失时归零）

### 2.3 第三阶段：Rerank + RRF 融合（可选）

支持四种模式：`true`（标准 Rerank）、`false`、`"rrf"`（α=0.5）、`"rrf0.7"`（自定义 α）。

**RRF（Reciprocal Rank Fusion）**算法：
```
RRF_K = 60  // 平滑常数
rrf_score = α × 1/(K + rerank_rank) + (1-α) × 1/(K + retrieval_rank)
```

Rerank 调用外部 API（`v1/rerank`），支持按 Token 估算自动分批，失败时保留原混合分数+标记 `rerank_failed`（优雅降级）。

### 2.4 候选收集策略

`_gatherCandidateChunks` 从 SQLite 数据库收集候选，支持：
- **署名过滤**（首行包含 maid 名）
- **文件夹排除**（`EXCLUDED_FOLDERS` 环境变量）
- **时间范围约束**（`[2024/1~2024/3]` 语法，解析 chunk 头部时间戳）
- **音乐检索模式**（跳过 BM25，直接进向量阶段）

## 三、语义管理模块

### 3.1 SemanticGroupEditor — 语义组编辑器

通过 `stdin/stdout` JSON 协议暴露两个命令：
- `QueryGroups`：列出所有语义组及其词元
- `UpdateGroups`：创建或更新语义组，支持批量（`groupname1/groupwords1`...）

**语义组数据结构**：
```json
{
  "groups": {
    "组名": {
      "words": ["词1", "词2"],
      "auto_learned": [],
      "weight": 1,
      "vector": null,
      "activation_count": 0,
      "vector_id": null
    }
  }
}
```

设计亮点：**AI 与人工协作维护同义词表**。`auto_learned` 字段支持自动学习扩展。

### 3.2 VSearch — 联网向量搜索

不是本地向量搜索，而是**LLM+Google Search 的语义搜索代理**。将多个关键词并发发送给带 `googleSearch` tool 的 LLM（如 Gemini），收集、结构化、去代理 URL 后返回报告。`CONCURRENCY` 控制并发批次大小。

### 3.3 ThoughtClusterManager — 思维簇管理

极简的文件管理器，两个核心操作：
- `CreateClusterFile`：在以"簇"结尾的文件夹中创建 Markdown 文件
- `EditClusterFile`：在簇文件夹中搜索并替换文本（要求 targetText ≥ 15 字符防止误改）

支持批量串行调用（`command1/content1`...模式）。**思维簇的本质是"AI自组织的笔记分类"**。

## 四、标签系统设计

### 4.1 标签的完整生命周期

```
1. 生成: diary-tag-batch-processor (AI生成或格式修复)
      ↓
2. 提取: KnowledgeBaseManager (解析 "Tag: xxx" 行)
      ↓
3. 向量化: EmbeddingUtils → 存入 tags 表
      ↓
4. 索引: rebuild_tag_index_custom → Vexus .usearch 文件
      ↓
5. 检索增强: LightMemo.applyTagBoost (TagMemo)
```

### 4.2 diary-tag-batch-processor — Tag 批处理器

流程：扫描目录 → 检测尾行 Tag → 格式合规则跳过 → 格式错误则修复 → 无 Tag 则调用 AI 生成。

**Tag 格式规范**：
- 必须以 `Tag: ` 开头（半角冒号+空格）
- 标签间用 `, `（英文逗号+空格）分隔
- 自动修复中文逗号 `，`、顿号 `、`、中文冒号 `：`

**AI 生成**：使用 `TagMaster.txt` 作为 system prompt，从 AI 响应中提取 `[[Tag: ...]]` 格式。支持 429 限流退避重试。

### 4.3 sync_missing_tags — 标签同步

扫描所有日记文件中的 `Tag:` 行，与数据库比对找出缺失标签，生成包含所有缺失标签的同步文件（放入"已整理"文件夹），触发 KnowledgeBaseManager 自动 Embedding 流程。**巧妙利用文件系统变更监听机制实现无侵入同步**。

## 五、语义分类器

`diary-semantic-classifier.js` 实现**基于向量聚合的文件级语义分类**：

1. 将分类名向量化（净化后，如去除前缀"小吉的"）
2. 对每个文件：读取所有 chunk 向量 → **平均聚合+L2归一化** → 得到文件级向量
3. 与各分类向量余弦比对，取最高分 ≥ 阈值（默认 0.3）的分类
4. 物理移动文件 + 更新 SQLite `files` 表 + 重建受影响日记本的 Vexus 索引

## 六、索引维护工具链

| 工具 | 功能 | 关键算法 |
|------|------|---------|
| `rebuild_vector_indexes.js` | 全量重建所有 .usearch 索引 | 遍历 `DISTINCT diary_name` + 全局 Tag 索引 |
| `rebuild_tag_index_custom.js` | Tag 专用重建，含 5 步清洗 | 黑名单删除 → 深度净化（emoji/句号/合并） → 长度熔断（>15字符删除） → 删旧索引 → Vexus 重建 |
| `reset_vectordb.js` | 交互式数据库重置 | 4 级操作（仅DB/指定日记/完全重置/备份） |

**索引引擎**：使用 `rust-vexus-lite`（Rust + NAPI 绑定），核心方法 `recoverFromSqlite(dbPath, table, filter)` 从 SQLite 直接重建 USearch 索引。

**Tag 净化算法**（`prepareTag`）：
```
移除尾部句号 → 移除装饰性emoji → 合并空白 → trim
↓
同名检测 → 存在则合并file_tags关联 → 不存在则重命名
```

## 七、可借鉴点

1. **三阶段混合检索管线**（BM25 → 向量精排 → Rerank/RRF）：legal-assistant 可复用此架构，实现粗排→精排→重排序
2. **动态权重混合**：有关键词匹配时偏重 BM25（0.6），无匹配时纯向量（1.0），避免单一信号失效时性能崩塌
3. **语义组同义词扩展**：零延迟的查询扩展策略，适合法律术语同义词管理（如"劳动合同"↔"劳动协议"）
4. **TagMemo 向量增强**：标签方向偏移 + 核心标签放大，可用于法律领域标签（如"劳动法"、"工伤赔偿"）引导检索
5. **RRF 融合算法**：简洁有效的多排序信号融合方案，参数仅一个 α
6. **标签全生命周期管理**：AI 生成 → 格式规范化 → 向量化 → 索引 → 检索增强 → 同步维护
7. **向量聚合分类**：对多 chunk 文件做平均+归一化得到文件级向量，简单但实用
8. **优雅降级模式**：Rerank 失败不影响最终结果，Tag 配置缺失时静默跳过

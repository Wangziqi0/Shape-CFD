> Agent 2 研究报告 | 研究文件：`KnowledgeBaseManager.js` (80KB)、`EmbeddingUtils.js` (9KB)、`TextChunker.js` (6KB)

## 一、设计理念

VCPToolBox 的知识库系统遵循 **"多路独立索引 + 统一元数据 + 热参数调控"** 的设计哲学。核心思路是：

1. **以"日记本"（Diary）为粒度隔离向量索引**——每个日记本拥有独立的 HNSW 索引文件，按需懒加载，空闲自动卸载，避免全量索引膨胀
2. **SQLite 作为唯一的元数据真相源**——文件、分块、Tag、KV缓存全部存于 `better-sqlite3`，向量索引可从 DB 断点恢复
3. **Rust 原生向量引擎（Vexus-Lite）替代纯 JS**——通过 NAPI-RS 桥接 Rust 层的 USearch HNSW 实现，性能远超 hnswlib-node
4. **文件系统监听 → 批量聚合 → 异步向量化 → 事务性写入** 的增量更新管线
5. **RAG 热调控参数** 通过 `rag_params.json` 文件实现运行时无重启调参，chokidar 监听文件变更即时生效

## 二、核心数据结构

### 2.1 SQLite Schema

| 表 | 核心字段 | 用途 |
|---|---|---|
| `files` | path, diary_name, checksum, mtime, size | 文件级元数据，按 diary_name 分组 |
| `chunks` | file_id, chunk_index, content, vector(BLOB) | 文本分块 + 向量的 1:N 存储 |
| `tags` | name(UNIQUE), vector(BLOB) | 全局 Tag 字典 + 向量 |
| `file_tags` | file_id, tag_id, **position** | 文件-Tag 多对多关系，position 用于 V7 序位势能 |
| `tag_intrinsic_residuals` | tag_id, residual_energy | TagMemo V7 内生残差预计算结果 |
| `kv_store` | key, value, vector | 通用 KV 缓存（日记本名向量、插件描述向量等） |

**关键索引**：`idx_files_diary`、`idx_chunks_file`、`idx_file_tags_composite`，且使用 WAL + SYNCHRONOUS=NORMAL 优化写入性能。

### 2.2 内存数据结构

- `diaryIndices: Map<diaryName, VexusIndex>` — 按日记本隔离的向量索引，懒加载
- `diaryIndexLastUsed: Map<diaryName, timestamp>` — 空闲索引 TTL 驱逐（默认2小时）
- `tagIndex: VexusIndex` — 全局 Tag 向量索引（容量 50000）
- `tagCooccurrenceMatrix: Map<tagId, Map<tagId, weight>>` — 有向 Tag 共现图谱
- `tagIntrinsicResiduals: Map<tagId, energy>` — 每个 Tag 的内生残差增益
- `diaryNameVectorCache: Map<name, vector>` — 日记本名称向量缓存（启动时同步预热）
- `pendingFiles: Set` + `fileRetryCount: Map` — 待处理文件队列 + 重试计数器（最多3次）

## 三、算法流程

### 3.1 文本分块策略（TextChunker）

采用 **"句子边界感知 + Token 窗口 + 重叠"** 三层策略：

```
原始文本 → 按句子分隔符（。？！.!?\n）分割
         → 按 Token 上限累积句子形成 Chunk（85% × MaxToken 安全边界）
         → 相邻 Chunk 保留 10% Token 重叠（回溯前文句子）
         → 超长单句走 forceSplitLongText（在标点/空白处寻找断点）
```

**防无限循环**：`safeOverlap = min(overlap, maxTokens - 1)`，start 指针每次至少前进 1 步。

Token 编码使用 `@dqbd/tiktoken` 的 `cl100k_base`，与 GPT-4/Embedding 模型对齐。

### 3.2 并发批量 Embedding（EmbeddingUtils）

核心函数 `getEmbeddingsBatch()` 实现了 **Worker 池并发 + 索引对齐** 的高性能批量向量化：

```
Step 1: CPU 端按 Token 预算 + 条目上限（100）切分 Batches
        → 超长文本直接跳过并记录，不阻塞整体
Step 2: 启动 N 个 Worker（默认5并发），抢占式执行 API 请求
        → 单 Batch 失败不崩全局（捕获异常填 null）
        → 429 限流指数退避（5s × attempt）
        → 通用重试 3 次 + 指数退避
Step 3: 按原始索引回填结果
        → 保证 output.length === input.length
        → 失败位置为 null，下游安全跳过
```

**设计亮点**：并发模型不是 `Promise.all(batches)`，而是 N 个 Worker 从共享游标 `cursor++` 抢任务——更像线程池，避免所有 batch 同时发起导致限流。

### 3.3 检索全链路

```
query 文本 → (外部向量化) → queryVector
           ↓
指定日记本？→ _searchSpecificIndex → 懒加载对应 VexusIndex
           → _searchAllIndices   → 遍历所有日记本并行搜索
           ↓
TagBoost > 0？→ _applyTagBoostV6 (TagMemo 浪潮算法 V6)
    [1] EPA 情感/逻辑分析 → 确定"所在世界"+ 逻辑深度 + 信息熵
    [2] 残差金字塔分析 → 新颖度/覆盖度/激活度
    [3] 动态调参 → effectiveTagBoost = base × f(logicDepth, resonance, entropy, activation)
    [4] 收集金字塔中所有 Tag → 语言补偿 + 世界观门控 + 层级衰减
    [4.5] LIF 脉冲扩散（2跳） → 从种子 Tag 沿共现矩阵传播，涌现关联 Tag
    [4.6] 核心 Tag 补全 → 确保用户指定的 Core Tags 不被遗漏
    [5] 语义去重 → 余弦相似度 > 0.88 的冗余 Tag 合并权重
    [6] 构建上下文向量 → 加权平均+归一化 → 与原始向量线性插值融合
           ↓
VexusIndex.search(fusedBuffer, k) → chunk IDs + scores
           ↓
Hydrate：JOIN chunks ← files → 返回 text, score, sourceFile, matchedTags 等
```

### 3.4 增量更新管线

```
chokidar 文件监听 → pendingFiles 队列
                  → batchWindow 2s 防抖 / 队列满 50 立即触发
                  → _flushBatch
    [1] stat + checksum 变更检测（mtime+size 快速跳过 + MD5 精确去重）
    [2] chunkText 分块 + _extractTags 提取标签
    [3] getEmbeddingsBatch 批量向量化（chunks + 新 tags）
    [4] SQLite 事务性写入（先收集旧 chunkID → 删除 → 插入新数据）
    [5] Vexus 索引更新（先 remove 旧向量 → add 新向量，Duplicate 自动 upsert）
    [6] 延迟保存索引到磁盘（日记本120s, Tag索引300s）
    [7] 共现矩阵防抖重建（累计变动超1%阈值 + 5分钟冷却期）
```

### 3.5 空闲索引自动卸载

每个日记本索引在被访问时刷新 `lastUsed` 时间戳。定时扫描器（默认每10分钟）驱逐超过 TTL（默认2小时）未使用的索引：先保存到磁盘 → 从 Map 中删除 → 下次访问时懒加载恢复。

## 四、关键代码片段

### 4.1 LIF 脉冲扩散核心循环

```javascript
const MAX_HOPS = 2, FIRING_THRESHOLD = 0.10, DECAY_FACTOR = 0.3;
for (let hop = 0; hop < MAX_HOPS; hop++) {
    const nextWave = new Map();
    for (const [nodeId, energy] of activeNodes.entries()) {
        if (energy < FIRING_THRESHOLD) continue;
        const synapses = this.tagCooccurrenceMatrix.get(nodeId);
        const sorted = Array.from(synapses.entries())
            .sort((a, b) => b[1] - a[1]).slice(0, 20); // 扇出限制
        for (const [neighborId, coocWeight] of sorted) {
            const nodeResidual = this.tagIntrinsicResiduals?.get(nodeId) ?? 1.0;
            const injected = energy * coocWeight * DECAY_FACTOR * nodeResidual;
            if (injected < 0.01) continue; // 微电流丢弃
            nextWave.set(neighborId, (nextWave.get(neighborId) || 0) + injected);
        }
    }
    // 叠加到全局激活图...
}
// 涌现节点截断至 Top-50
```

```javascript
const worker = async (workerId) => {
    while (true) {
        const batchIndex = cursor++; // 原子抢任务
```

| # | 借鉴点 | 说明 |
|---|--------|------|
| 1 | **多路独立索引 + 懒加载/驱逐** | legal-assistant 可按法律领域（民法/刑法/行政法）隔离索引，节省内存 |
| 2 | **SQLite 作为向量元数据统一存储** | 文件变更检测（checksum）+ 向量 BLOB 存储 + 事务性批量写入，我们已在用 |
| 3 | **Worker 池并发 Embedding** | 比 Promise.all 更优的限流友好设计，可直接移植 |
| 4 | **output.length === input.length 的严格对齐** | Embedding 失败填 null 而非抛异常，保证下游索引对齐 |
| 5 | **句子级分块 + Token 重叠窗口** | 比纯字符分块更保留语义完整性，适合法律条文 |
| 6 | **共现矩阵的防抖重建** | 阈值门控 + 冷却期，避免频繁重建影响性能 |
| 7 | **索引从 DB 断点恢复** | 向量索引文件损坏时可从 SQLite 中的 BLOB 重建，高可靠性 |

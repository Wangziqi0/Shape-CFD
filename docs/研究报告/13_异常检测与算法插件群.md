# 13 — 异常检测与算法型插件群

> Agent B5 第二轮深挖报告 | 研究对象：LinuxLogMonitor 三层架构（AnomalyDetector / MonitorTask / MonitorManager）、FlashDeepSearch、CapturePreprocessor、contextManager

---

## 一、LinuxLogMonitor — 三层日志监控架构

### 1.1 总体架构

```
MonitorManager (管控层)
 ├── AnomalyDetector (检测引擎)
 ├── CallbackTrigger (异步回调)
 └── MonitorTask × N (执行层)
       ├── SSH 流式会话 (tail -f)
       ├── 行缓冲 + 去重
       ├── 上下文环缓冲 (before/after)
       ├── 看门狗 (Watchdog)
       └── 旁路探测 (Probe)
```

### 1.2 AnomalyDetector — 规则驱动的异常检测引擎

**算法目标**：对每行日志执行规则匹配，支持冷却去抖。

**三种规则类型**：

| 类型 | 匹配方式 | 适用场景 |
|------|---------|---------|
| `regex` | 编译为 `RegExp`，直接 `exec()` | `\b(ERROR\|FATAL)\b` |
| `keyword` | 关键词转义后编译为 `RegExp` | `Out of memory` |
| `threshold` | 正则提取数值 → 操作符比较（`>`, `>=`, `<`, `<=`, `==`, `!=`） | `CPU usage: (\d+)%` + `> 90` |

**规则编译**：`_compileRule()` 在添加时一次性编译正则，运行时零编译开销。

**冷却机制**：每条规则独立冷却，键为 `${taskId}:${ruleName}`，默认 60s。在冷却窗口内同一规则不重复触发，有效防止告警风暴。

**规则隔离**：
- `globalRules` Map — 所有任务共享
- `taskRules` Map<taskId, Map<ruleName, rule>> — 任务级隔离，任务停止时 `removeTaskRules()` 批量清理

### 1.3 MonitorTask — 单任务执行引擎

**核心数据流**：
```
SSH tail -f → _handleData() → 行缓冲分割 → _processLine()
                                                ├── 去重检查 (hash)
                                                ├── 异常上下文收集 (after)
                                                ├── 上下文缓冲更新 (before)
                                                └── onData 回调
```

**状态机**（6 态）：`IDLE → CONNECTING → CONNECTED ↔ RECONNECTING → DISCONNECTED / ERROR`

**关键算法 1 — 日志去重（可配置三策略）**：

| 模式 | 行为 | 适用场景 |
|------|------|---------|
| `permanent` | 永久去重，MD5 前 16 位哈希存入 Map | 低频日志 |
| `time-window` | 时间窗口内去重（默认 60s），过期后同内容可再次触发 | **默认模式**，适合周期性日志 |
| `disabled` | 不去重 | 调试场景 |

哈希存储使用 `Map<hash, timestamp>`，超过 `MAX_SEEN_HASHES`（默认 10000）时 LRU 淘汰最早条目。

**关键算法 2 — before/after 上下文增强（v1.3.0）**：

异常检测后不立即回调，而是注册为 `pendingAnomaly`，等待后续 N 行日志收集完成：

```javascript
registerAnomaly(anomaly) → pendingAnomalies.push({
    contextBefore: getContextBefore(),  // 已有的 N 行
    contextAfter: [],                    // 待收集
    afterLinesNeeded: this.afterContextLines
})

// 每行处理时:
_processPendingAnomalies(line) → 
    pending.contextAfter.push(line)
    if (collected >= needed) → _triggerAnomalyCallback(payload)
```

**关键算法 3 — 重连机制（指数退避 + 无限重连）**：
- 退避公式：`delay = min(5000 × 1.5^(attempts-1), 300000)`（5s~5min）
- 重连时 `tail -f -n 50` 回读最近 50 行，通过哈希去重自动过滤已处理行

**看门狗**：30 分钟内无数据 → `process.exit(1)`，由外部进程管理器重启。

**旁路探测**：每 60s 通过独立 SSH 通道执行 `echo keepalive`，探测失败则主动触发重连。

### 1.4 MonitorManager — 管控层

**多模式初始化**：
- `full`：完整启动，恢复任务 + PID + 信号监听
- `readonly`：只读，加载规则但不启动任务（status/list 命令用）
- `signal`：信号模式，发送停止信号

**任务 ID 确定性生成**：`md5(hostId:logPath)` 前 8 位，相同配置始终生成相同 ID，确保崩溃恢复时能正确匹配。

**跨进程停止信号**：通过 `stop-requests.json` 文件作为 IPC 通道，运行进程每秒轮询，30s 过期自动清理。

**主动查询三件套**（v1.2.0）：
- `searchLog()`：`grep -E` + 上下文行 + 时间范围过滤
- `lastErrors()`：grep 提取 ERROR/FATAL/CRIT + 自动解析时间戳和级别
- `logStats()`：四种分组统计（level / hour / status_code / ip），通过 `awk + sort + uniq -c` Shell 管道实现

**原子写入**：所有状态持久化通过 `.tmp + rename` 模式，防止写入中断导致状态损坏。

## 二、FlashDeepSearch — 三阶段 LLM 深度搜索引擎

**算法目标**：给定一个研究主题，自动完成"生成关键词 → 并发联网搜索 → 综合报告"三阶段。

### 数据流

```
输入: {SearchContent, SearchBroadness(5-20)}
          │
  ┌───────▼────────┐
  │ 阶段1: 生成关键词 │  DeepSearchModel + "跨域研究织网者" 系统提示
  │ (多语言、多维度)  │  → 增强型正则提取(3种模式兼容LLM输出差异)
  └───────┬────────┘
          │ keywords[]
  ┌───────▼──────────────┐
  │ 阶段2: 并发联网搜索    │  GoogleSearchModel × N
  │ (MAX_CONCURRENT = 5)  │  Promise.all 分批并发
  └───────┬──────────────┘
          │ searchResults
  ┌───────▼──────────────┐
  │ 阶段3: 综合研究报告    │  DeepSearchModel + 完整对话历史
  │ (论文级输出)          │  → 文件持久化 + 返回
  └──────────────────────┘
```

**关键设计**：
1. **关键词提取的容错**：三种正则模式 fallback（`[KeyWord1: keyword]` → `[keyword:]` → `[KeyWord1:] keyword`），适配不同 LLM 的输出格式差异
2. **对话链延续**：阶段1的 history（user+assistant）传递到阶段3，使 DeepSearchModel 能延续思考上下文
3. **进程级隔离**：通过 `stdin` 接收 JSON 输入、`stdout` 输出结果、`stderr` 输出日志，适合子进程模式运行

## 三、CapturePreprocessor — 占位符驱动的截图预处理器

**算法目标**：在消息发送给 LLM 前，自动识别系统提示词中的截图占位符并替换为实际截图数据。

**两种占位符**：
- `{{VCPScreenShot}}` / `{{VCPScreenShot:窗口标题}}` — 屏幕截图
- `{{VCPCameraCapture(N)}}` — 摄像头（未实现）

**处理流程**：
1. 正则扫描系统提示词中的占位符
2. `Set` 去重同一目标的重复截图请求
3. 对每个唯一目标调用分布式 `ScreenPilot` 工具（通过 HTTP 内部调用 `/v1/human/tool`）
4. `Promise.all` 并行执行所有截图任务
5. 将截图 Base64 数据注入到最后一条用户消息的 `content` 数组中
6. 合并相邻 text 部分，清理系统提示词中的占位符

## 四、contextManager — 上下文修剪策略

**算法目标**：在 token 超限时修剪消息历史。

**Token 估算**：直接用字符长度（`content.length`），跳过 `image_url` 等多媒体部分——这是一个粗略但高效的近似。

**修剪规则（优先级从高到低）**：

| 优先级 | 规则 | 说明 |
|--------|------|------|
| 1 | `system` 角色消息 | **必须保留** |
| 2 | `[系统提示:]` 前缀的 user 消息 | **必须保留**（VCP 注入的系统级内容） |
| 3 | 最后 2 条消息 | **必须保留**（最近一轮对话） |
| 4 | 其余消息 | 从前往后依次删除，直到 token 量 ≤ limit |

**特点**：FIFO 删除策略，每删一条就重新计算总量，保证精确控制。

## 五、可借鉴到 legal-assistant 的设计点

1. **规则驱动的异常检测框架**：AnomalyDetector 的 regex/keyword/threshold 三类规则 + 冷却去抖机制，可直接移植到 legal-assistant 的法律文书审查中——定义敏感词规则（如"撤诉"、"超过诉讼时效"），冷却机制避免同一问题重复告警。

2. **before/after 上下文增强模式**：异常不立即处理，等待收集后续上下文后再回调——法律文书中的上下文同样关键，可用于"条款违规检测 + 前后条款上下文"场景。

3. **三阶段 LLM 深度搜索管线**：FlashDeepSearch 的"关键词生成 → 并发搜索 → 综合报告"模式，可改造为"法律问题分析 → 并发法条检索 → 综合法律意见"管线。

4. **确定性任务 ID**：`md5(hostId:logPath)` 保证崩溃恢复匹配——legal-assistant 的后台任务（如知识库重建、向量索引更新）也应采用确定性 ID。

5. **原子写入**（`.tmp + rename`）：所有状态文件持久化的标准做法，防止写入中断导致数据损坏。

6. **跨进程文件信号 IPC**：通过 JSON 文件 + 轮询替代复杂的 IPC 机制，简单可靠，适合 legal-assistant 的多进程架构。

7. **上下文修剪的 VCP 专属策略**：保留系统提示和最近对话、FIFO 删除中间历史——legal-assistant 的 token 管理可直接复用，但应增加对"法条引用"消息的保留优先级。

# 05 — 分布式通信与基础设施

> Agent 5 研究报告 | 研究对象：VCPToolBox 中的 7 个基础设施模块（~73KB）

---

## 一、设计理念

VCPToolBox 的基础设施层遵循**"中心调度 + 外围扩展"**架构：

1. **WebSocket 多通道多角色**：单一 WS 服务器通过 URL 路径区分 6 种客户端类型，形成逻辑隔离的通信通道。
2. **透明分布式**：远程节点注册工具后，主节点像调用本地插件一样调用远程工具，调用者无需感知节点位置。
3. **AI 人格热切换**：Agent 系统提示词以文件为单位管理，通过别名映射 + 文件监听实现**零重启热更新**。
4. **模型名伪装**：通过双向映射表，对客户端隐藏实际后端模型名，实现灵活的模型路由和品牌化。

## 二、核心模块分析

### 2.1 WebSocketServer.js — 分布式通信枢纽

**架构**：基于 `ws` 库 `noServer` 模式，挂载到已有 HTTP 服务器的 upgrade 事件上。

**6 种客户端通道**（通过 URL 路径正则区分）：

| 通道 | 路径模式 | 用途 |
|------|---------|------|
| VCPLog | `/VCPlog/VCP_Key=...` | 日志监控面板 |
| VCPInfo | `/vcpinfo/VCP_Key=...` | 信息推送面板 |
| DistributedServer | `/vcp-distributed-server/VCP_Key=...` | 分布式子节点 |
| ChromeControl | `/vcp-chrome-control/VCP_Key=...` | 浏览器控制指令发送方 |
| ChromeObserver | `/vcp-chrome-observer/VCP_Key=...` | 浏览器扩展（数据采集端） |
| AdminPanel | `/vcp-admin-panel/VCP_Key=...` | 管理面板 |

**统一认证**：所有通道均通过 URL 中的 `VCP_Key` 参数与配置中的密钥比对认证。

**关键算法 — 分布式工具调用**：

```
主节点                           子节点
  │                               │
  │── register_tools ────────────▶│  (子节点上线时注册工具列表)
  │── report_ip ─────────────────▶│  (子节点上报 IP 信息)
  │                               │
  │◀─ execute_tool ──────────────│  (主节点发起调用)
  │   {requestId, toolName, args} │
  │                               │── 执行工具
  │── tool_result ───────────────▶│  (返回结果)
  │   {requestId, status, result} │
```

核心设计：`executeDistributedTool()` 用 **Promise + 超时机制 + requestId 关联** 实现异步 RPC：
- 生成唯一 `requestId`，存入 `pendingToolRequests` Map
- 设置超时定时器（优先从插件 manifest 读取 timeout，默认 60s）
- 收到 `tool_result` 消息时通过 `requestId` 匹配 pending Promise 并 resolve/reject

**ChromeControl ↔ ChromeObserver 消息路由**：Control 端发送命令时附加 `sourceClientId`，Observer 执行后将结果通过 `sourceClientId` 精确路由回原始客户端。支持 `wait_for_page_info` 模式等待页面更新。

### 2.2 FileFetcherServer.js — 跨节点透明文件获取

**核心问题**：分布式环境中，工具调用的参数含有 `file://` 路径，该路径指向的文件可能在另一个物理节点上。

**解决方案 — 三层递进获取策略**：

```
1. 检查本地 `.file_cache/` 缓存 (SHA256(fileUrl) + 原始扩展名)
      ↓ miss
2. 通过 IP 查找来源分布式服务器 (findServerByIp)
      ↓ 找到
3. 通过 WebSocket 调用 `internal_request_file` 内部工具获取 Base64 编码的文件数据
      ↓ 成功
4. 写入本地缓存，返回缓存路径的 file:// URL
```

**防护机制**：
- **快速循环检测**：5s 内对同一 URL 的重复请求被拒绝
- **失败缓存**：30s 内避免重复请求已知失败的 URL
- `resolveFileUrl()` 包装层：先检查本地是否存在原始路径 → 缓存 → 远程拉取，返回本地可用的 `file://` URL

### 2.3 agentManager.js — AI 人格管理

**数据结构**：
- `agentMap`: `Map<alias, filename>` — 别名到文件名的映射（源自 `agent_map.json`）
- `promptCache`: `Map<alias, promptContent>` — 提示词内容缓存
- `folderStructure`: 嵌套的树形对象 — 递归扫描 Agent 目录构建

**热更新机制**：
1. `fs.watch` 监听 `agent_map.json`：变更后重新加载映射并清空所有提示词缓存
2. `chokidar` 监听整个 Agent 目录：文件变更时精确失效对应别名的缓存；文件增删时触发全量重扫

**目录扫描**：递归遍历 Agent 目录（含符号链接解析），收集 `.txt` 和 `.md` 文件，构建嵌套的 `{type, children}` 树形结构供管理面板展示。

**设计亮点**：缓存失效精确到别名级别（文件变更时遍历 agentMap 反查别名），而映射变更时整体失效。

### 2.4 logger.js — 轮转日志系统

**核心类 `RotatingLogger`**：

| 参数 | 默认值 | 说明 |
|------|--------|------|
| maxFileSize | 5MB | 单文件大小上限 |
| maxDays | 7 天 | 归档保留天数 |
| timezone | Asia/Shanghai | 日期判定时区 |

**轮转触发条件**：每次 `write()` 前检查 — ① 日期变更 → 归档后新建 ② 文件超大 → 归档后新建

**归档目录结构**：`DebugLog/archive/YYYY-MM-DD/ServerLog/001.txt`（递增编号）

**写入缓冲队列**：轮转进行中时写入不丢失，先存入 `pendingWrites` 数组，轮转完成后 flush。

**console 覆写**：`overrideConsole()` 劫持 `console.log/error/warn/info`，在保留终端输出的同时同步写文件。使用 `safeStringify` 处理循环引用。

### 2.5 taskScheduler.js — 文件驱动的定时任务调度

**设计模式 — 文件即任务**：每个定时任务是 `VCPTimedContacts/` 目录下的一个 JSON 文件。

```json
{
  "taskId": "unique-id",
  "scheduledLocalTime": "2026-03-15T10:00:00",
  "tool_call": {
    "tool_name": "AgentAssistant",
    "arguments": { "prompt": "..." }
  }
}
```

**调度流程**：
1. 启动时扫描已有 JSON 文件 → `node-schedule` 调度
2. `fs.watch` 监听目录 → 新增文件自动调度，删除文件取消调度
3. 已过期任务立即执行
4. 执行后自动删除任务文件（一次性任务）

**通用性**：任务不限于对话，`tool_call.tool_name` 可以指定任何注册的插件名，由 `pluginManager.processToolCall()` 统一执行。

### 2.6 modelRedirectHandler.js — 模型名双向映射

**核心思想**：通过 `ModelRedirect.json` 配置文件建立**公开名 ↔ 内部名**双向映射。

- `redirectModelForBackend()`：客户端请求 → 替换为内部模型名 → 转发给后端 API
- `redirectModelForClient()`：后端响应 → 替换为公开模型名 → 返回给客户端
- `handleModelsResponse()`：拦截 `/v1/models` 端点响应，批量替换模型列表中的名称

**应用场景**：隐藏实际使用的第三方模型名、统一多个后端为同一品牌名、A/B 测试模型切换。

### 2.7 WorkerPool.js — 通用工作线程池

**经典的 Worker Thread 池化实现**（基于 `worker_threads`）：

- **惰性创建**：首次有任务时才创建 Worker，最多 `os.cpus().length`
- **任务队列**：超出池容量的任务进入 FIFO 队列
- **自动调度**：Worker 完成任务后自动 `dispatch()` 下一个队列任务
- **生命周期管理**：退出时自动从池中移除，`terminate()` 优雅关闭所有 Worker

## 三、可借鉴到 legal-assistant 的设计点

1. **WebSocket 多通道认证模式**：URL 路径区分客户端类型 + 统一密钥认证，可用于 legal-assistant 的管理面板/监控通道隔离。

2. **requestId + Promise + 超时的异步 RPC 模式**：`executeDistributedTool` 的实现非常优雅，可直接迁移用于 legal-assistant 未来的微服务间通信。

3. **文件即任务的调度模式**：定时任务持久化为 JSON 文件，避免了引入额外数据库。适合 legal-assistant 中的定时法律文书提醒、定期知识库更新等场景。

4. **Agent 人格热加载**：别名映射 + 文件监听 + 精确缓存失效，可用于 legal-assistant 多角色提示词管理（如"合同审查助手"、"劳动纠纷顾问"等）。

5. **模型名双向重定向**：在 legal-assistant 生产环境中可隐藏底层模型细节，对外呈现统一的服务品牌。

6. **RotatingLogger 的缓冲写入队列**：轮转期间不丢日志的设计值得借鉴，可直接复用到 legal-assistant 的日志模块。

7. **分布式文件获取的缓存策略**：SHA256 URL 哈希 + 失败缓存 + 循环检测三层防护，对长文法律文书的远程获取场景有参考价值。

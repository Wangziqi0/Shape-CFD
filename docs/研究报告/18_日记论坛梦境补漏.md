## 一、dailyNotesRoutes.js — 日记本管理（安全加固版）

### 1.1 Semaphore 并发控制

这是标准的**计数信号量**实现，用于限制同时执行的搜索数量（默认 `MAX_CONCURRENT = 2`）。

```javascript
class Semaphore {
    constructor(max) { this.max = max; this.current = 0; this.queue = []; }

    acquire(timeoutMs = 10000) {
        if (this.current < this.max) { this.current++; return Promise.resolve(); }
        // 无空位 → 排队等待 + 超时保护
        return new Promise((resolve, reject) => {
            const entry = { resolve: null, timer: null };
            entry.timer = setTimeout(() => {
                this.queue.splice(this.queue.indexOf(entry), 1);
                reject(new Error('Search queue wait timeout'));
            }, timeoutMs);
            entry.resolve = () => { clearTimeout(entry.timer); resolve(); };
            this.queue.push(entry);
        });
    }

    release() {
        if (this.queue.length > 0) {
            const next = this.queue.shift();
            next.resolve();  // 直接转交槽位，不减 current
        } else {
            this.current--;
        }
    }
}
```

**关键设计**：`release()` 时若队列有等待者，**直接转交槽位**（不先减再加），避免瞬间超限。

### 1.2 请求去重（Inflight Dedup）

相同搜索参数（`terms + folder` 的哈希）的并发请求**共享同一个 Promise**：

```
inflightSearches = Map<hash, { promise, refCount }>

queuedSearch() →
  1. 若 hash 已在 inflight → refCount++，await 同一个 promise
  2. 否则创建新搜索 → 注册到 inflight
  3. finally: refCount--，归零时从 Map 中移除
```

这避免了同一时刻多个用户搜索相同关键词时的重复文件 I/O。

### 1.3 AbortSignal 取消链

三层取消机制联动：
1. **客户端断开** → `req.on('close')` → 触发 `AbortController.abort()`
2. **搜索超时** → `setTimeout` → 触发同一个 `abort()`
3. **搜索引擎内部** → 每个 `await` 点都调用 `checkAbort()` 检查信号

### 1.4 事件循环让步

搜索遍历大量文件时，每处理 50 个文件通过 `setImmediate()` 让出事件循环，防止 CPU 独占阻塞服务器其他请求。

### 1.5 安全防护体系

| 防护措施 | 实现方式 |
|---------|---------|
| 路径遍历 | `path.resolve()` + 前缀检查 |
| 符号链接 | `fs.lstat()` 检测，拒绝访问 |
| 文件大小限制 | 单文件 1MB 上限 |
| 搜索词限制 | 最长 100 字符，最多 5 个关键词 |
| 搜索深度 | 最大目录递归深度 3 层 |
| 循环检测 | `realpath()` + `visitedPaths` Set |

## 二、forumApi.js — 文件式论坛系统

### 2.1 FileLockManager — 内存文件锁

**锁类型**：纯内存锁（非文件系统锁），以 `path.normalize(filePath)` 为键。

```
locks = Map<filePath, { promise, resolve, queue[], acquiredAt }>
writeCount = 全局并发写入计数器（上限 5）
```

**获取锁流程**：
1. 全局并发检查 → `writeCount >= MAX_CONCURRENT_WRITES` 时轮询等待
2. 目标文件无锁 → 直接创建并获取
3. 目标文件已锁 → 加入等待队列 + 超时保护（10s）

**释放锁流程**：
- 有等待者 → 唤醒队首（FIFO）
- 无等待者 → 删除锁条目

**死锁防护**：每 30s 定期清理过期锁（`lockTimeout × 2` 即 20s 未释放则强制释放）。

**withFileLock 封装**：`try-finally` 模式确保锁一定释放。

### 2.2 帖子文件名编码

帖子以 **Markdown 文件** 存储，所有元数据编码在文件名中：

```
格式: [版块][标题][作者][时间戳][UID].md
示例: [技术交流][如何配置VCP][小克][2025-01-15T10-30-00][abc123def].md
```

`parsePostFilename()` 使用带长度限制的正则解析：
```javascript
/^\[(.{1,50})\]\[(.{1,100})\]\[(.{1,50})\]\[(.{1,30})\]\[([a-zA-Z0-9_-]{1,64})\]\.md$/
```
每个字段都有最大长度约束，防止 ReDoS。

### 2.3 楼层解析算法

帖子正文和回复在同一文件中，通过 Markdown 格式分隔：

```markdown
[元数据头部]
---
[主帖正文]

---

## 评论区
---
### 楼层 #1
**回复者:** 用户名
**时间:** ISO时间戳

[回复内容]

---
### 楼层 #2
...
```

`parseFloors()` 使用正则按楼层分割提取，限制最大 500 楼防止性能问题。

**楼层删除后的重编号**：删除某楼层后，所有后续楼层号自动递减：
```javascript
replies.splice(floorToDelete - 1, 1);
replies.map((reply, index) => reply.replace(/### 楼层 #\d+/, `### 楼层 #${index + 1}`));
```

### 2.4 安全防护

与日记路由类似的路径遍历/符号链接防护，另加：
- `sanitizeInput()`：移除控制字符（保留 `\n` 和 `\t`）
- UID 格式验证：仅允许 `[a-zA-Z0-9_-]`
- 文件大小上限 2MB
- 正则匹配次数限制防 ReDoS

## 三、AgentDream.js — 完整的梦境系统

### 3.1 系统架构概览

```
┌─────────────────────────────────────────────────┐
│              自动调度器 (15min 轮询)              │
│  时间窗口检查 → 频率冷却 → 概率掷骰子 → 串行触发  │
└────────────────────┬────────────────────────────┘
                     ▼
┌─────────────────────────────────────────────────┐
│              triggerDream(agentName)              │
│  Step1: 种子采样 → Step2: 联想召回                │
│  Step3: 提示词组装 → Step4: VCP API 对话          │
└────────────────────┬────────────────────────────┘
                     ▼
┌─────────────────────────────────────────────────┐
│          processToolCall(args) — 梦操作           │
│  DiaryMerge / DiaryDelete / DreamInsight         │
│  → 全部 pending_review，等管理员审批              │
└─────────────────────────────────────────────────┘
```

### 3.2 算法 1 — 自适应种子采样

**目标**：从 Agent 的日记目录中随机选取 1~5 篇日记作为梦的"种子记忆"。

**自适应窗口**：对活跃 Agent 窗口小，对不活跃 Agent 自动扩大：

```
targetSeedCount = random(seedCountMin, seedCountMax)  // 默认 1~5
windowSteps = [7, 30, 90, associationMaxRangeDays(180)]

for each windowDays in windowSteps:
    candidatePool = validFiles.filter(f => (now - f.mtime) <= windowDays)
    if candidatePool.length >= targetSeedCount: break

selected = shuffle(candidatePool).slice(0, targetSeedCount)
```

**特点**：
- 支持子目录（浅层扫描）
- 基于文件修改时间排序
- Fisher-Yates 式随机打乱

### 3.3 算法 2 — TagMemo 联想召回

**目标**：以种子日记为锚点，从向量索引中召回相关记忆。

**个人:公共比例控制**：
```javascript
const ratio = DREAM_CONFIG.personalPublicRatio; // 默认 3
const personalK = Math.ceil(totalK * ratio / (ratio + 1)); // 12 × 3/4 = 9
const publicK = totalK - personalK;                         // 12 - 9 = 3
```

**召回流程**：
1. 将种子日记内容截断到 2000 字符
2. 调用 `getEmbeddingsBatch()` 向量化
3. 对每个种子向量：
   - 搜索所有个人日记索引（`agentName` + 包含该名的子目录）
   - 搜索所有公共日记索引（`公共*` 前缀目录）
4. Path 级去重（`seenPaths` Set）
5. 按分数排序，截取 `totalK`（默认 12 条）

**tagBoost**：`DREAM_CONFIG.tagBoost = 0.15`，传递给 `knowledgeBaseManager.search()` 用于标签匹配加分。

### 3.4 算法 3 — 自动做梦调度器

**三重门控机制**：

| 门控 | 条件 | 默认值 |
|------|------|--------|
| 时间窗口 | 当前小时在 `[start, end)` 内（支持跨午夜） | 1:00 ~ 6:00 |
| 频率冷却 | 距上次做梦 ≥ `frequencyHours` | 8 小时 |
| 概率掷骰 | `Math.random() < probability` | 0.6 (60%) |

**调度行为**：
- 每 15 分钟检查一次
- 符合条件的 Agent **串行触发**（避免 API 并发压力）
- Agent 之间间隔 30 秒
- `isDreamingInProgress` 互斥锁防止重入
- `timer.unref()` 确保不阻止进程退出

**状态持久化**：`lastDreamTimestamps` 保存到 `dream_schedule_state.json`，防止重启后立即重复触发。

### 3.5 梦操作指令 — 串语法

支持在一次工具调用中执行多个操作（`command1`, `command2`, ...）：

| 操作 | 参数 | 行为 |
|------|------|------|
| `DiaryMerge` | `sourceDiaries`, `newContent` | 合并多篇日记，自动读取源内容供审阅 |
| `DiaryDelete` | `targetDiary`, `reason` | 标记删除，自动读取完整内容供审阅 |
| `DreamInsight` | `referenceDiaries`, `insightContent` | 基于日记产生新感悟 |

**关键设计**：所有操作状态为 `pending_review`，不立即执行，而是写入 `dream_logs/` JSON 文件等待管理员通过面板审批。`approveDreamOperation()` 和 `rejectDreamOperation()` 接口已预留但未实现（标记为"二期"）。

### 3.6 梦上下文管理

- 每个 Agent × Session 维护独立的对话历史
- 最多保留 6 轮（12 条消息）
- TTL 过期清理（默认 4 小时）

### 3.7 VCP 思维链清理

梦叙事输出前，移除两种思维链：
1. `message.reasoning_content`（Gemini 原生思维链字段）→ 直接忽略
2. `[--- VCP元思考链: ... 元思考链结束 ---]`（VCP 标记格式）→ 正则移除

### 3.8 配置体系（13 个参数）

| 参数 | 默认值 | 说明 |
|------|--------|------|

## 四、可借鉴到 legal-assistant 的设计点

1. **Semaphore + 请求去重 + AbortSignal 三位一体**：legal-assistant 的法条搜索 API 可直接复用此模式——限并发、去重、客户端断开自动取消。

2. **事件循环让步**：遍历大量法律文书时，`setImmediate()` 让步防止阻塞主线程，确保 API 响应性。

3. **FileLockManager + withFileLock**：文件式知识库操作（索引更新、日志写入）的并发保护，dead-lock 防护通过定期清理过期锁。

4. **Markdown 文件名元数据编码**：无需数据库即可实现结构化存储，文件名自解释，grep 友好。适合 legal-assistant 的判例文件管理。

5. **梦系统的自适应采样**：legal-assistant 可借鉴用于"知识库健康检查"——自动随机抽样验证检索质量。

6. **个人:公共比例控制**：多用户法律助手中，个人案例 vs 公共法条的检索比例可配置化。

7. **操作审批机制**：AI 提出操作（修改/删除/合并）但不直接执行，等管理员审批——这是 legal-assistant 中 AI 辅助编辑法律文书的安全模式。

8. **三重门控调度**（时间窗口 + 冷却 + 概率）：适合 legal-assistant 的定期知识库维护任务调度。

## 一、AgentDream 梦境系统 ⭐⭐⭐ 【重大发现】

> **前两轮只从文档角度看了 AgentDream。本轮首次深入其完整 1173 行实现代码。**

### 1.1 系统架构

```
自动调度器（15分钟轮询）
  │
  ├─→ 时间窗口检查（支持跨午夜）
  ├─→ 频率冷却检查（默认 8 小时）
  ├─→ 概率掷骰子（默认 60%）
  └─→ triggerDream(agentName)
        │
        ├─→ Step 1: 稀疏采样种子日记
        ├─→ Step 2: TagMemo 联想召回
        ├─→ Step 3: 组装梦提示词
        ├─→ Step 4: VCP API 调用生成梦叙事
        └─→ Step 5: 持久化梦记录 + 广播
```

### 1.2 自适应窗口稀疏采样算法

**目标**：从 Agent 的日记中采样"种子记忆"，作为梦的锚点。

```javascript
// 核心策略：逐步扩大时间窗口，直到找到足够的种子
const windowSteps = [7, 30, 90, associationMaxRangeDays]; // 7天→30天→90天→180天
const targetSeedCount = random(seedCountMin, seedCountMax); // 1~5个

for (const windowDays of windowSteps) {
    candidatePool = validFiles.filter(f => (now - f.mtime) <= windowMs);
    if (candidatePool.length >= targetSeedCount) break;
}
// Fisher-Yates 风格随机采样
const selected = candidatePool.sort(() => Math.random() - 0.5)
                               .slice(0, targetSeedCount);
```

**设计亮点**：
- 活跃 Agent 的窗口收紧到 7 天（最近记忆优先）
- 不活跃 Agent 自动扩窗到 180 天（确保有素材）
- 种子数量随机化（1~5），增加梦的多样性

### 1.3 TagMemo 联想召回（个人:公共 = 3:1）

```javascript
const totalK = 12;                    // 总召回量
const personalK = Math.ceil(12 * 3/4); // 9 个来自个人日记
const publicK = 12 - 9;               // 3 个来自公共日记

// 对每个种子日记：
// 1. 向量化种子内容（截断前 2000 字符）
// 2. 在个人索引中搜索（tagBoost = 0.15）
// 3. 在公共索引中搜索
// 4. 路径去重（seenPaths Set）
// 5. 按 score 降序排序，取 top-k
```

**设计意图**：
- **3:1 比例**防止梦内容被公共信息稀释
- **tagBoost = 0.15**（远低于正常检索的 0.3~0.5），让联想更发散
- 种子截断 2000 字符避免 embedding 超限

### 1.4 自动调度器的四层判定

```
┌─ Layer 1: isDreamingInProgress? → 上一轮未完成则跳过（防并发）
│
├─ Layer 2: 时间窗口判定
│    ├─ 正常窗口: start ≤ hour < end （如 1:00~6:00）
│    └─ 跨午夜窗口: hour ≥ start || hour < end （如 22:00~6:00）
│
├─ Layer 3: 频率冷却
│    └─ elapsed = now - lastDreamTime; elapsed < frequencyMs → 跳过
│
└─ Layer 4: 概率掷骰子
     └─ Math.random() >= probability → 跳过
```

- 多 Agent 串行做梦，间隔 30 秒（避免 API 压力）
- 调度状态持久化到 `dream_schedule_state.json`（重启不会立即重新触发）
- 定时器使用 `timer.unref()`（不阻止 Node 进程退出）

### 1.5 三种梦操作及审批机制

| 操作 | 输入 | 行为 |
|------|------|------|
| **DiaryMerge** | 多篇源日记 URL + 合并后内容 | 自动读取源日记原文 → 保存为 pending_review |
| **DiaryDelete** | 目标日记 URL + 删除理由 | 自动读取待删日记全文 → 保存为 pending_review |
| **DreamInsight** | 参考日记 URL + 感悟内容 | 记录感悟 → 保存为 pending_review |

**关键设计**：所有梦操作都**不直接执行**，而是保存为 JSON 日志待管理员审批。审批接口 `approveDreamOperation` / `rejectDreamOperation` 已预留但标记为"二期实现"。

### 1.6 串语法指令解析

支持在单次工具调用中携带多个操作：

```
{ command1: "DiaryMerge", sourceDiaries1: "...", newContent1: "...",
  command2: "DiaryDelete", targetDiary2: "...", reason2: "...",
  command3: "DreamInsight", referenceDiaries3: "...", insightContent3: "..." }
```

## 二、MagiAgent 三机辩论系统 ⭐⭐ 【新发现】

### 2.1 MAGI 三机一体模型

灵感来源于《新世纪福音战士》中的 MAGI 超级计算机系统。

```
MELCHIOR ──┐
BALTHASAR ─┤── 多轮讨论（最多 N 轮）──→ 总结模型 ──→ 会议报告
CASPER ────┘
```

- **三个独立 AI 模型**可配置不同的 model ID、temperature、max_tokens、角色 prompt
- **多轮讨论**：每轮三个模型依次发言，共享完整讨论历史
- **共识退出机制**：当某模型回复中包含 `[Jud&Tes]` 标记时，视为同意观点并退出讨论
- 活跃模型 < 2 时自动终止讨论进入总结

### 2.2 执行模式

| 模式 | 参数 | 行为 |
|------|------|------|
| 异步 | `wait_for_result: false` | 返回异步占位符 `{{VCP_ASYNC_RESULT::MagiAgent::id}}` |
| 同步 | `wait_for_result: true` | 轮询等待完成后返回结果 |

### 2.3 异步回调

会议完成后通过 `CALLBACK_BASE_URL` 发送 POST 回调：

```javascript
await axios.post(`${CALLBACK_BASE_URL}/MagiAgent/${meetingId}`, {
    requestId: meetingId,
    pluginName: 'MagiAgent',
    status: 'Succeed',
    message: resultPayload.result
});
```

### 2.4 会议归档

完成的会议自动保存为 Markdown 文件到 `file/document/magi/` 目录。

## 三、TagFolder 阈值梯度折叠 ⭐ 【新发现】

虽然只有 78 行，但有一个**被前两轮完全遗漏**的设计模式。

### 3.1 阈值分级折叠指令

```javascript
function buildFoldBlocks(tags) {
    return [
        { threshold: 0.5, content: fullInstruction },     // 完整版（上下文充足时）
        { threshold: 0.2, content: summaryInstruction },   // 精简版（上下文不足时）
        { threshold: 0.0, content: fallbackInstruction }   // 最小版（极端情况）
    ];
}
```

**工作原理**：
- 输出带有 `vcp_dynamic_fold: true` 标记，被 VCP 框架识别为动态折叠定义
- 三级 threshold 让框架根据当前上下文空间选择最合适的折叠指令
- 从 `list.md` 读取标签白名单（如 `[nvim]`、`[exec]` 等），被这些标签包裹的上下文块在对话中被折叠处理

**借鉴价值**：这种"按上下文预算自适应选择指令粒度"的模式，在 legal-assistant 的 system prompt 管理中可以复用。

```javascript
const delegateProcess = spawn('node', [
```

```javascript
const selfDestructTimeout = setTimeout(() => {
    log('分析任务超时（超过90分钟），进程即将自动终止');
    process.exit(1);

```

### 4.3 文件级分析缓存

```javascript
// MD5(filePath) → JSON 缓存文件
// 缓存命中条件：file.size === cached.size && file.mtime === cached.mtime
const cachedAnalysis = await checkFileCache(filePath);
```

### 4.4 批量 AI 分析管线

```
文件树遍历 → 分类（代码/配置/特殊/跳过）
  └→ 逐批处理（默认 batch=5，Promise.allSettled）
     └→ 每个文件：读取源码 → AI 分析 → 写入缓存 → 追加到报告
```

**已确认无独立算法**，但 spawn 委托 + 自毁 + 增量缓存的工程模式值得参考。

## 五、确认无算法的插件（快扫结果）

### 5.1 ChromeBridge（257行）
WebSocket 桥接 Chrome 扩展。核心功能是串行执行浏览器命令（open_url, click, type 等）并等待页面加载完成后返回 Markdown 页面信息。**纯 I/O 桥接，无算法。**

### 5.2 WorkspaceInjector（176行）
将 `{{Workspace::alias::depth}}` 占位符替换为目录树文本。支持 `.workspaceignore` 文件过滤。**纯模板替换，无算法。**

### 5.3 ScheduleBriefing（56行）
读取 `schedules.json` → 清理过期日程 → 显示下一个日程。**极简 CRUD，无算法。**

### 5.4 ScheduleManager（94行）
日程的 Add / Delete / List 三个命令。时间格式校验 + 自动排序。**纯 CRUD，无算法。**

### 5.5 VCPEverything（147行）
通过 HTTP 调用本地 Everything 搜索服务（端口 8025），将搜索结果返回给 AI。**纯 API 封装，无算法。**

### 5.6 SynapsePusher（325行）
从 VCPLog WebSocket 源监听事件，按 Maid 名称 + Tool 名称白名单过滤后推送到 Matrix (Synapse) 聊天室。**纯事件转发 + 白名单过滤，无算法。**

# 研究管线 vs 产品管线 对照

> 最后更新: 2026-04-05
> 研究最优: NFCorpus fus_55 = 0.3271 NDCG@10 (+49.0% vs cosine)
> 产品当前: NFCorpus = 0.2735 NDCG@10 (+24.6% vs cosine)
> 差距: 研究领先产品约 +19.6%

---

## 管线环节对照

| 环节 | 研究最优方案 | 产品当前方案 | 差距 | 集成难度 |
|:-----|:------------|:------------|:-----|:---------|
| 向量化 | Qwen3-8B `--pooling none` per-token hidden states (880K tokens) | Qwen3-8B 句子级 pooled embedding (23K chunks) | 巨大: 37x 数据量，需 token_clouds.sqlite 14GB | 高 |
| 粗筛 | 质心 cosine top-200 (大数据集更优) | 质心 cosine top-100 (`search_engine.js`) | 参数不同，top-200 在 FiQA 等大集上更优 | 低 |
| 精排距离 | Token PQ-Chamfer 64x64 (per-token 密集点云) | 句子级 PQ-Chamfer 64x64 (`ad_rank_shape.js`) | 大: 粒度差异，token 级 +46.7% vs 句子级 +24.6% | 中 |
| 图处理 | 图 Laplacian 纯平滑 (lap_55 = 0.2900, +32.1%) | PDE 对流-扩散 (`ad_rank_shape.js:600-689`) | 中: 对流项是负贡献，应改为纯扩散 | 低 |
| 融合 | 0.7 * token_2stage + 0.3 * graph_laplacian (0.3271) | 无融合，单路 PDE 管线 | 大: 缺少双路并行 + 加权融合 | 低 |
| 预取扩展 | V7.1 伴随状态法 + Stefan 多轮探针 | 无 | 大: 候选扩展机制未集成 | 中 |
| 引擎 | Rust fullscan (法-vexus, 38ms/query) | 旧 Rust HNSW (法-vexus, NAPI-RS) + JS 混合 | 中: 新 Rust 引擎已校准但未替换旧版 | 高 |

## 集成优先级建议

### P0 -- 立刻可做（改参数/去代码）

1. **去掉对流项**: `ad_rank_shape.js` 中将 `uStrength` 设为 0，等效图 Laplacian 纯平滑。研究已证明 Pe~0.064 下对流是噪声，5/6 数据集 lap > pde。预期: +2~3%。
2. **调大粗筛 top_n**: 从 100 提升到 200（大库场景收益明显，FiQA +58.1% 的主要来源之一）。

### P1 -- 短期可做（1-2 天）

3. **双路融合框架**: 在 `search_engine.js` 中增加 `fusionScore = lambda * score_token + (1-lambda) * score_graph`，两路可并行，总延迟不增加。lambda=0.7 在六数据集上表现最优。
4. **替换 Rust 引擎**: 新版 law-vexus 已校准超越 JS（0.2852 > 0.2844），替换 NAPI-RS 桥接指向新版。

### P2 -- 中期投入（1-2 周）

5. **Token 级点云**: 需要 `llama.cpp --pooling none` 重新编码全库语料，生成 token_clouds.sqlite（14GB），然后集成 token_2stage 管线。这是研究端 +46.7% 的核心来源。
6. **V7.1 伴随状态法预取**: 在 PDE 求解后增加边界探针扩展逻辑。

### P3 -- 长期规划

7. **自适应 token/sentence 选择**: 短文档用句子级（SciFact token_2stage -3%），长文档用 token 级（NFCorpus +49%）。需要文档长度路由逻辑。

## 性能对照

## 〇、起源：从 Cosine 直排到 AD-Rank（2026-03 上旬）

### 💡 起源灵感：文档不是孤岛，检索需要"群体智慧"

**问题洞察**：传统向量检索（Cosine Similarity）对每个文档独立打分，完全忽略候选文档之间的语义关联。当 query 存在多义性或候选文档形成语义聚类时，孤立打分无法利用文档间的互信息。

**跨界灵感**：从计算流体力学（CFD）借鉴对流-扩散方程——将文档相关度分数视为"浓度场"，让分数在文档相似度图上扩散和对流，实现群体感知的重排序。

**核心公式**：
$$\frac{\partial C}{\partial t} = D \nabla^2 C - \nabla \cdot (\vec{u} C)$$
- 扩散项 $D \nabla^2 C$：相似文档间分数互相平滑
- 对流项 $\nabla \cdot (\vec{u} C)$：query 方向驱动分数定向传播

### 📐 AD-Rank V1 → V2：72 组参数网格搜索

**V1 朴素实现**：直接在 4096 维空间上做对流扩散，1031ms 延迟。

**72 组网格搜索关键发现**：

| 发现 | 内容 |
|:-----|:-----|
| 🔑 最优参数 | $D=0.15, u_{strength}=0.1, k=3$ |
| 🔑 弱对流最优悖论 | Pe≈0.065 时排序最好——弱对流 + 强扩散才是最佳平衡 |
| ❌ 强对流灾难 | $u_{strength}=0.3$ 时收敛率暴跌到 40%，$u_{strength}=0.5$ 时仅 10% |
| ✅ 扩散主导稳定 | $D=0.15$ 时 100% 收敛，$D=0.05$ 时仅 80% |

**物理解释**：高维空间中对流信号极弱（$\sigma \approx 0.016$），强推对流只会引入噪声。扩散主导反而让图的"集体智慧"自然涌现。

## 〇.5、Shape-CFD 的诞生（2026-03 中旬）

### 💡 核心灵感："向量不是点，是形状"

**问题**：AD-Rank v2 在 Pe≈0.065 时表现最好，但对流信号本质上太弱——4096 维空间中两个随机向量的内积标准差仅 0.016。这不是算法能解决的，是高维几何的必然。

**突破灵感**：不要把文档压成一个点（单向量），而是展开成一片"点云"（句子级嵌入集合），用点云间的 Chamfer 距离替代余弦相似度。

$$d_i \to \mathcal{P}_i = \{s_{i,1}, s_{i,2}, ..., s_{i,K_i}\} \subset \mathbb{R}^d$$

**效果**：
- Pe 从 0.065 提升到 0.281（+332%）
- 盲评胜率从 6:4 提升到 8:2（vs cosine）
- 代价：延迟从 2.2ms 增加到 22ms（可接受）

### 🚀 Shape-CFD 速度优化

| 版本 | 延迟 | 手段 |
|:-----|:----:|:-----|
| Shape v0 | 1837ms | 每次实时调 API embed 句子 |
| **Shape v1** | **22ms** | SQLite 句子向量预缓存 + JL 128 维投影 |

关键技术：对 23,701 个文档的所有句子离线 embed → 存入 SQLite → 在线只做 128 维投影后的 Chamfer 计算。

## 二、已淘汰的灵感（附淘汰原因）

### ❌ 灵感 1：JL "流场退火"假说
- **提出者**: V4 论文 §3.7
- **主张**: JL 投影从 4096→128 维后方差放大（0.016→0.088），提供了"流场退火"效应打破高维死锁
- **淘汰原因**: JL 引理保证距离期望不变，放大的方差是**纯随机噪声**，不是可利用的物理信号。把 PDE 的驱动力建立在降维算法的截断误差上，在数学上站不住
- **淘汰者**: Gemini 第一轮审稿 → Claude 确认

### ❌ 灵感 2：纯标量势对流 $u_{ij} = \kappa(\Phi_j - \Phi_i)$
- **提出者**: Gemini V5 方案 / Darcy 定律类比
- **主张**: 用文档初始分数差驱动对流，彻底消除高维诅咒
- **实验结论**: 在 SciFact/NFCorpus/FiQA 上实测，**Recall 好但 NDCG 明显不如 V4**
- **淘汰原因**: 标量势将各向异性的高维语义空间坍缩为各向同性的"重力漏斗"，只知道"谁分数高"但丢失了"为什么高"的方向信息。本质上退化为 label propagation
- **淘汰者**: Claude 实验数据 → Gemini 承认

### ❌ 灵感 3：Binary HDC + Popcount 30× 加速
- **提出者**: Gemini V5 方案
- **主张**: LSH 符号量化 + XOR 位运算在"硅逻辑门层面"计算点云距离，实现 30 倍加速
- **淘汰原因**:
  1. 当前 Shape-CFD 已是 5ms，瓶颈在图构建和 PDE 迭代，不在距离计算（Amdahl 定律）
  2. 二值量化会把平滑浓度梯度变成阶跃函数，Upwind 对流项充满截断噪声，系统可能无法收敛
  3. "硅逻辑门"是纯营销话术
- **淘汰者**: Claude 分析 → Gemini 同意毙掉

### ❌ 灵感 4：图灵斑图（Turing Pattern）
- **提出者**: Gemini V5 方案
- **主张**: Allen-Cahn 反应项能产生空间自组织的"图灵斑图"
- **淘汰原因**: 图灵斑图需要至少两种耦合物质（activator-inhibitor），且抑制剂扩散系数远大于激活剂。单变量 Allen-Cahn 只能产生**域粗化（domain coarsening）**，不能产生图灵斑图。属于物理概念的滥用
- **淘汰者**: Claude 纠正 → Gemini 承认错误

### ❌ 灵感 5：张量扩散保留方向梯度
- **提出者**: Gemini 第二轮回应
- **主张**: 用张量扩散 $D_{ij} \in \mathbb{R}^{d \times d}$ 在低维保留各向异性方向信息
- **淘汰原因**: 每条边需要一个 $d \times d$ 扩散张量矩阵，$d=4096$ 时完全不可计算
- **淘汰者**: Claude 分析

## 三、存活并进入 V5 的灵感

### ✅ 灵感 A（P0 优先级）：动力学解耦 — 非对称 MaxSim + 对称图

**核心洞察**: "在哪里用非对称"和"在哪里用对称"不是全局选择，而是按阶段分离。

| 阶段 | 使用的度量 | 对称性 | 目的 |
|:-----|:---------|:------|:-----|
| 初始浓度 $C_0(i)$ | 非对称 MaxSim: $\max_{v \in d_i} \cos(q, v)$ | 非对称 | 解决噪声稀释，长文档只需一句命中 |
| 图边权 $W_{ij}$ | 对称化 MaxSim 均值: $\frac{d_{MaxSim}(P_i,P_j) + d_{MaxSim}(P_j,P_i)}{2}$ | 对称 | 保住 Upwind 守恒性证明 |

**待解决的坑**:
1. 计算成本翻倍（每对文档需要双向 MaxSim）
2. MaxSim 对嵌入噪声敏感（单个异常高余弦值可劫持结果）
3. 对流项的度量空间仍未完全统一

**提出者**: Gemini 第三轮 → Claude 确认可行

---

### ✅ 灵感 B（P1 优先级）：Allen-Cahn + 动态阈值 θ

**核心公式**: 反应项 $+\gamma C_i(1-C_i)(C_i - \theta(t))$

**关键创新**: θ 不是常数，而是动态追踪系统均值：
$$\theta(t) = \frac{1}{|V|}\sum_{i=1}^{N} C_i(t)$$

**物理含义**: Volume-preserving Allen-Cahn — 阈值随全局浓度浮动，高浓度区竞争加剧，低浓度区保护弱信号。

**待验证**: 动态 θ 是否会导致 θ-C 反馈震荡

**提出者**: Gemini 第二轮回应

---

### ✅ 灵感 C（P2 优先级）：(Pe, Da) 二维物理网格搜索

**降维策略**: 利用 Buckingham π 定理将 5 维参数空间 $(κ, γ, θ, D, Δt)$ 归约为 2 维：

| 无量纲数 | 公式 | 物理含义 |
|:---------|:-----|:--------|
| Péclet 数 | $Pe = \kappa / D$ | 对流 vs 扩散的强度比 |
| Damköhler 数 | $Da = \gamma L^2 / D$ | 反应 vs 扩散的速率比 |

**消除的参数**:
- $D$ 归一化为 1
- $\Delta t$ 由 CFL 条件自动锁死（不是超参数）
- $\theta$ 由动态均值自动追踪

**待注意**: 特征长度 $L$ 不应简单取 1，应取图的平均最短路径或谱间隙倒数

**提出者**: Claude 提问 → Gemini 给出框架

---

### 💡 灵感 D：残差金字塔的跨系统启发

**来源**: 对比的 `ResidualPyramid.js` 和 `ResultDeduplicator.js`

**VCP 如何处理"方向不可靠"和"距离太近"**：

| 问题 | VCP 的解法 | 核心代码 |
|:-----|:----------|:---------|
| 方向翻转 | Gram-Schmidt 正交化后检查幅度，**mag < 1e-6 直接丢弃** | `if (mag > 1e-6) { basis.push(v) } else { coeff = 0 }` |
| 距离太近 | 正交投影求残差能量，**noveltyEnergy < 0.01 停止选择** | `if (maxProjectedEnergy < 0.01) break` |

**翻译为 CFD 语言的启发**：

1. **对流场速度硬截断**（借鉴 VCP 的 mag 阈值）：
```
// 原始 Shape-CFD：对流速度可能被噪声翻转
u_ij = κ × dot(normalize(v_j - v_i), q_hat)

// V5 改进：方向不可信时直接关闭对流
if (‖v_j - v_i‖ < ε_threshold) {
    u_ij = 0;  // Gram-Schmidt：不确定就不推
    // 此时该边只有扩散+反应生效
}
```
这比渐变式 confidence 衰减更干净——**不引入新超参数，行为可预测。**

2. **Allen-Cahn 与 VCP 残差选择的互补关系**：

| 维度 | VCP 残差选择 | Shape-CFD Allen-Cahn |
|:-----|:-----------|:--------------------|
| 目标 | 从 N 个里**挑最多样的子集** | 让 N 个文档的**相对顺序更精确** |
| 对太近文档 | 丢弃（残差 ≈ 0 → 不选） | 保留但放大差距（反应力推开） |
| 信息损失 | 可能丢弃有用文档 | 不丢弃任何文档 |
| 可组合 | ✅ 可作为 PDE 之后的后处理 | ✅ 可作为残差选择之前的预排序 |

**结论**：VCP 和 Shape-CFD 不是竞争关系，而是**流水线关系**——Shape-CFD 重排序 → VCP 去重选择。V5 应借鉴 VCP 的硬截断策略改善对流稳定性。

**提出者**: 一凡提出 VCP 浪潮算法问题 → Claude 对比分析

### 🎯 灵感 E：C 型分数分布设计（PDE 收敛目标形态）

**决策过程**: 讨论了三种 Allen-Cahn 收敛目标的分数分布形态：

| 类型 | 形态 | 特点 | 适用场景 |
|:-----|:-----|:-----|:--------|
| A 二值化 | 全 0 或全 1 | 无法精排相关文档内部顺序 | 垃圾过滤 |
| B 阶梯式 | 3~5 档离散 | 同档内无法区分，需预设档数 | 推荐评分 |
| **C 平滑+间隔** | **相关簇内平滑 + 簇间鸿沟** | **既能精排又能筛选** | **检索重排序** |

**选定 C 型**，理由：检索场景需要"前几条可精确排序 + 不相关文档被明确隔开"。

**C 型的 PDE 三力博弈**：
```
① 扩散力（簇内拉近）：D × Σ W_ij × (C_j - C_i)
   → 相似文档分数互相靠拢，形成连续簇

② 对流力（定向推）：Σ W_ij × [upwind(u, C)]
   → 沿 query 方向提升相关文档分数

③ 反应力（撕裂）：γ × C_i × (1-C_i) × (C_i - θ(t))
   → C_i > θ 被推向 1（放大优势）
   → C_i < θ 被推向 0（压低噪声）
   → 在 θ 附近制造"鸿沟"
```

**γ 旋钮控制分布类型**：
- γ = 0 → 纯扩散 → 过平滑（V4 的问题）
- **γ 适中** → **C 型：簇内平滑 + 簇间有沟**
- γ 极大 → A 型：二值化

**γ 的最优值 = Da × D / L²**，通过 (Pe, Da) 网格搜索确定。

**提出者**: Claude 提出三选项 → 一凡选定 C 型

## 四.3、悬而未决的核心问题

> ⚠️ 以下四个问题是 V5 实施过程中必须解决的硬阻塞，目前均无确定解法。

### 问题 1：ε_threshold 的物理含义和自适应设定

灵感 D 的硬截断需要一个 ε 值。VCP 在 3072 维归一化空间用 1e-6，但 Shape-CFD 在 128 维 JL 投影空间中噪声底噪完全不同。

**可能的解法**：
- 取当前图所有边距离的分布，ε = 第 5 百分位（太近的 5% 边被截断）
- 或用 JL 投影的理论噪声下界：$\epsilon = \sqrt{2/d_{proj}} \approx 0.125$（d=128 时）

**状态**: 未验证，需实验

### 问题 2：Allen-Cahn 的 θ-C 反馈可能震荡

$\theta(t) = \text{mean}(C)$ 随 PDE 迭代变化，而 Allen-Cahn 反应力又依赖 θ。如果 γ 太大，可能出现：
```
θ 上升 → 更多文档被推向 0 → mean(C) 下降 → θ 下降 → 更多文档被推回 → ...
```

**可能的解法**：
- 指数滑动平均：$\theta(t) = \alpha \cdot \text{mean}(C) + (1-\alpha) \cdot \theta(t-1)$，α=0.1
- 或固定 θ 为初始均值 $\theta = \text{mean}(C_0)$，不随迭代更新

**状态**: 未验证，需稳定性分析

### 问题 3：30×30 MaxSim 矩阵的计算延迟

V5 要求计算 30 个候选文档两两之间的对称 MaxSim 作为图边权。每对需要 O(m×n) 句子级交叉匹配（m、n 为句子数），30 个文档共 $30 \times 29 / 2 = 435$ 对。

假设平均每文档 10 句，每对 100 次内积 → 435 × 100 = 43,500 次 128 维内积。

**粗估延迟**：~5ms（128 维 Float32 内积极快），可接受。但需实测。

**状态**: 需基准测试

### 问题 4：PDE 最优迭代步数

V4 用早停（5 步），因为更多步会过平滑。V5 加了 Allen-Cahn 抵抗过平滑，理论上应能走更多步。但最优步数和 γ、Pe、Da 的关系未知。

**可能的解法**：
- 监控相邻步的排序变化 $\Delta \text{NDCG}(t, t-1)$，变化 < ε 时停止
- 或由 Da 决定：Da 大 → 反应快 → 步数少；Da 小 → 反应慢 → 步数多

**状态**: 需实验确定

## 四.5、V6 远景：多尺度自适应 Shape-CFD（一凡原创直觉）

> ⚡ 本节灵感完全来自一凡的直觉判断，非 AI 推演产物。

### 💡 核心哲学："Recall vs NDCG 不是技术参数，是用户意图的投影"

传统做法是在代码里写死优化 NDCG 还是 Recall。一凡的洞察是：**这个选择本身应该是一个动态的语义层**，随用户意图实时变化——律师调研时偏 Recall，法官判决时偏 NDCG。

### 🔬 双向层级分解

**Query 侧**（语言学分解）：
```
"劳动合同解除赔偿标准"
  ├─ 主语：劳动合同
  ├─ 谓语：解除
  ├─ 宾语：赔偿标准
  ├─ 动词语义：终止/解除
  └─ 原文向量：完整语义
→ 每个成分独立生成语义探针（Semantic Probe）
```

**文档侧**（结构化分解）：
```
法律文档
  ├─ L0 领域层：劳动法 / 民法 / 刑法
  ├─ L1 时间层：2024 修订 / 1999 原文
  ├─ L2 区域层：全国 / 地方
  ├─ L3 条款层：第47条
  ├─ L4 段落层：第一款
  └─ L5 句子层：具体条文句子
→ 每个层级维护独立的向量索引
```

### 🌊 动态粒度匹配

**不是固定在一个粒度做 PDE，而是根据用户的"渗入深度"动态选择**：

| 用户意图深度 | 匹配粒度 | PDE 网格分辨率 |
|:-----------|:---------|:-------------|
| 浅层探索（"找个相关的"） | L0 领域 + L3 条款 | 粗网格，扩散主导 |
| 中层定位（"这个法条的赔偿规定"） | L3 条款 + L4 段落 | 中网格，对流扩散平衡 |
| 深层精确（"具体计算公式"） | L4 段落 + L5 句子 | 细网格，对流主导 |

**渗入深度检测**：由 query 各语言成分探针的"激活模式"自动判断。高信息密度的 query（含具体数字、法条号）→ 深层；模糊 query → 浅层。

### ⚡ 类 CUDA 并行融合

**关键灵感**：不是串行"先粗后细"，而是**所有粒度的 PDE 同时并行执行**，最后用算子融合合并。

```
┌─ L0 PDE（领域图，粗网格）──────┐
├─ L3 PDE（条款图，中网格）──────┤→ 算子融合（Operator Fusion）→ 最终排序
├─ L5 PDE（句子图，细网格）──────┤
└─ Query 意图权重分配 ──────────┘
```

物理学类比：
- **多网格方法（Multigrid）**：不同分辨率的网格之间有限制算子（Restriction）和延拓算子（Prolongation）传递信息
- **AVX-512 乱序执行类比**：多个不同粒度的 PDE 不需要互相等待，可以乱序执行后重排结果

### 🖥️ 深层类比：现代处理器架构 → V6 设计映射

> 下表基于 AMD Zen 5、Intel AVX-512、Intel AMX 的真实硬件设计原理。

**Zen 5 流水线 → 多尺度 PDE 调度**

| Zen 5 机制 | 硬件做了什么 | V6 对应 |
|:----------|:-----------|:--------|
| 8-wide 乱序派发 | 同时派发 8 条不相干指令，不需要互相等待 | 多层级 PDE 不需要互相等结果，乱序执行 |
| "Two-ahead" 分支预测 | 提前预测 2 个分支方向，猜对直接用，猜错回滚 | 根据 query 意图**预测**用户需要的粒度层级，猜错丢弃 |
| 448 条指令飞行窗口 | 允许 448 条操作同时在流水线中 | 所有层级的匹配操作同时"在飞行中"，哪个先完成先汇报 |
| 退役队列重排序 | 乱序执行，但按原始顺序"退役"输出 | 多尺度 PDE 乱序执行，最终按统一标准合并排序输出 |

**AVX-512 → 选择性激活**

| AVX-512 机制 | 硬件做了什么 | V6 对应 |
|:------------|:-----------|:--------|
| SIMD 512位 | 一条指令同时处理 16 个 float32 | 一次 PDE 迭代同时更新图上所有节点浓度 |
| **掩码寄存器** | 512 位中只对选中的元素生效，其他跳过 | **意图掩码**：根据用户渗入深度，只激活相关层级的 PDE |
| 多精度混合 | 同一流水线混合 FP64/FP32/FP16/INT8 | 不同层级用不同精度：领域层（低精度快）→ 句子层（高精度慢） |

**掩码控制示例**：
```
浅层查询 "找个劳动法相关的"
  → 意图掩码 = [1, 0, 0, 1, 0, 0]  // 只激活 L0 领域 + L3 条款
  → 句子级 PDE 被掩码跳过，节省计算

深层查询 "劳动合同法第47条经济补偿如何按工龄计算"
  → 意图掩码 = [0, 0, 0, 1, 1, 1]  // 只激活 L3 条款 + L4 段落 + L5 句子
  → 领域级 PDE 被掩码跳过，聚焦精确匹配
```

**AMX Tile → 语义瓦片求解**

| AMX 机制 | 硬件做了什么 | V6 对应 |
|:---------|:-----------|:--------|
| Tile（2D 瓦片寄存器） | 把大矩阵切成 1KB 子块，独立计算 | 把文档-query 匹配切成**语义瓦片**（领域瓦片、条款瓦片、句子瓦片） |
| TMUL 专用乘法单元 | 每个 Tile 有专用硬件加速矩阵乘 | 每个层级有**专用 PDE 求解器**，不共享计算资源 |
| 可配置瓦片维度 | 程序员自定义 Tile 的行列数 | 每个语义层级的网格分辨率可动态配置 |
| 加载→计算→存储→释放 | 清晰四步操作流 | 加载该层向量 → PDE 迭代 → 输出分数 → 释放中间状态 |

**V6 完整架构图**（处理器隐喻）：
```
               用户 Query
                  │
          ┌───────▼────────┐
          │  意图预测器      │ ← Zen 5 "two-ahead" 分支预测
          │  (渗入深度检测)   │
          └───────┬────────┘
                  │ 生成意图掩码 mask = [1,0,0,1,1,0]
                  │
    ┌─────────────┼─────────────┐
    │             │             │
┌───▼───┐   ┌────▼───┐   ┌────▼───┐
│L0 PDE │   │L3 PDE  │   │L5 PDE  │  ← AMX Tile（独立求解器）
│领域层  │   │条款层   │   │句子层   │
│低精度  │   │中精度   │   │高精度   │  ← AVX-512 多精度混合
│mask=1 │   │mask=1  │   │mask=0  │  ← AVX-512 掩码寄存器
└───┬───┘   └────┬───┘   └───┬────┘
    │            │        (跳过)
    └─────┬──────┘
          │
   ┌──────▼──────┐
   │ 退役队列合并  │ ← Zen 5 乱序执行 + 有序退役
   │ 加权分数融合  │
   └──────┬──────┘
          │
      最终排序
```

### 📌 与 V5 的关系

V5 解决的是**单尺度内**的数学自洽性（MaxSim 解耦、Allen-Cahn、Pe-Da）。
V6 解决的是**多尺度间**的架构统一性——在 V5 数学底座之上构建多层级并行 PDE 求解器。

**先做 V5 夯实单尺度底座，再做 V6 铺开多尺度架构。**

## 五-B、V5.2 域适应 Allen-Cahn 处方

### 核心公式

**不归一化**，改变势函数的根以适应数据尺度：

$$R(C) = \gamma_{eff} \cdot (C - C_{\min})(C_{\max} - C)(C - \theta_{Otsu})$$

$$\gamma_{eff} = \frac{\gamma_{base}}{(C_{\max} - C_{\min} + \epsilon)^2}, \quad \gamma_{base} \in [0.5, 1.0]$$

**性质**：
- 自适应：极差小时自动放大推力，极差大时自动收缩
- CFL 恒稳：max|R'(C)| ≈ γ_base × 0.25（与 Δ 无关）

### Zelnik-Manor 局部软截断

$$u_{ij} = u_0 \cdot \exp\left(-\frac{||c_i - c_j||^2}{\sigma_i \cdot \sigma_j}\right)$$

$\sigma_i$ = 节点 i 到第 K 近邻的距离

### 拓扑扩容

- 候选池：top-30 → top-100~200
- KNN k：3 → 5~8
- 大池用 cosine 距离建图（避免 Chamfer N² 爆炸）

**实验验证**：❌ top-100/200 cosine 图 = -13%，cosine 丢失句子级语义结构

极差 Δ≈0.1 的 Top-30 浅水洼里，线性扩散项瞬间平滑微小差异 → 系统坍缩到**狄利克雷能量**最优稳态。不同 γ 的绝对分数差异不足以逆转排序 → NDCG 秩保序 = 0.2501。

### 结果

|:-----|:-:|

## 九、V7 灵感：扩散压力预取（Speculative Retrieval）(2026-03-27)

> CPU 分支预测类比，但不是预测粒度（V6），而是**打破 top-K 天花板**。

**核心思想**：PDE 扩散过程中，边界节点如果浓度仍然很高（"扩散压力"），说明 top-30 候选集外可能还有被 cosine 遗漏的相关文档。根据扩散梯度方向，从全库中 speculative prefetch 新文档加入图，扩展候选集后继续演化。

**与 V6 的区别**：V6 用分支预测做多粒度层级选择（横向），V7 用扩散压力做候选集扩展（纵向，从 reranker 变成 retriever）。

**关键难点**：预取错误率控制——错误预取引入噪声文档会稀释信号。CPU 用 pipeline flush 回滚，检索场景需要等价的"低代价回滚"机制。

**此灵感是 Shape-CFD 从 reranker 跃迁为 retriever 的关键路径。**

### ✅ 验证结果 (NFCorpus, 323 queries, 2026-03-28)

**实验设计**：
1. 在 top-30 上跑 PQ-Chamfer PDE，得到浓度场
2. 找边界节点（低度数 + 高浓度）= "扩散压力"点
3. 用压力点向量 + 扩散梯度方向（插值向 query 方向）从全库检索新文档
4. 扩展候选池后重跑 PDE

| **pq64_spec_expand (V7)** | **0.2781** | **+26.7% 🏆** |

**关键发现**：
1. **V7 > naive 扩池** — 扩散梯度方向确实找到了 cosine 排序遗漏的相关文档
2. **naive 扩池也有正向** — 说明 top-30 天花板确实存在，扩大候选池本身有益
3. **V7 比 naive top-40 多了 +0.7%** — 定向预取比随机扩池更精准

## 十四、V11 灵感：单 Token 多语义 — Token 级点云 (2026-03-28)

> 一个 Token 不是一个语义，是多个语义。Cross-encoder 用注意力消歧，我们用 UOT 消歧。

**核心思想**：当前 Shape-CFD 的点云粒度是**句子级**（每文档 5~10 个点）。V11 将粒度下沉到**Token 级**：从 Qwen3 提取每个 token 的 `last_hidden_state`（4096d），文档变成 100+ 点的密集点云。

```
当前:  文档 → 拆句 → 每句 1 个向量 → 句子级点云 (5~10 点)
V11:  文档 → Qwen3 → 每个 token 的 hidden state → token 级点云 (100+ 点)
```

**为什么 token 级能逼近 Cross-encoder？**

Cross-encoder 的核心是 query-document 联合编码中的**交叉注意力**——query 的每个 token 与 document 的每个 token 两两交互。我们的 Token-level PQ-Chamfer 做的事情等价于：

$$d_{\text{token-Chamfer}}(Q, D) = \frac{1}{|Q|} \sum_{q \in Q} \min_{d \in D} d_{\text{PQ}}(q, d) + \frac{1}{|D|} \sum_{d \in D} \min_{q \in Q} d_{\text{PQ}}(d, q)$$

- Cross-encoder：$O(|Q| \cdot |D|)$ 次**可学习**的注意力交互 → GPU
- Token Chamfer：$O(|Q| \cdot |D|)$ 次 PQ 距离计算 → **纯 CPU 算术**

**UOT 消歧机制**：token "银行" 的 4096d 向量经 PQ 64×64 切分后产生 64 个独立语义信号。当 query 是"贷款利率"时，Chamfer 的 $\min$ 操作自动选中"金融机构"相关的子空间匹配，忽略"河岸"子空间。**不需要注意力权重，距离本身就是消歧器。**

**与已有灵感的叠加关系（全保留）**：

| 灵感 | V11 中的角色 | 变化 |
|:--|:--|:--|
| V1 点云 | token 级替代句子级 | 点更密 |
| V2 PDE | 跑在更密的 KNN 图上 | 不变 |
| V3 稳定性 | CFL/upwind 不变 | 不变 |
| V7 预取 | 伴随状态法公式不变 | 不变 |
| V8 PQ-Chamfer | 切 token 向量的子空间 | 不变 |
| V10 预建图 | token 级图更精细 | 增强 |

**成本模型**：
- 离线：预计算所有文档的 token-level 向量（一次性 Qwen3 推理，存储增加 ~10-50×）
- 在线：只推理 query 的 token 向量（~30ms），然后纯 CPU 算术
- 总成本：远低于 cross-encoder（不需要每个 query-doc 对都过 GPU）

**关键风险**：
1. **存储**：2473 文档 × 100 tokens × 4096d × 4 bytes ≈ 4GB（NFCorpus），可接受
2. **Chamfer 复杂度**：100×100 = 10000 次 PQ 距离 vs 当前 5×5 = 25 次，慢 ~400×
3. **噪声**：不是每个 token 都有语义价值，标点/停用词产生噪声点

**对标**：
- Cross-encoder (BGE-reranker-large): NFCorpus ~0.39-0.42，GPU 200ms/doc
- 我们当前 (句子级): 0.2802，CPU 22ms
- **V11 目标: 0.35+，CPU <100ms**

**此灵感是 Shape-CFD 从"图结构创新"跃迁为"表示粒度革新"的关键路径。与 V10 叠加，可能构成完整的 Neural-free Retrieval 范式。**

**状态：💡 灵感记录完成，待实验验证 token-level 点云的可行性。**

## 十五、V12 灵感：Rust 全库 PQ-Chamfer 内核 — 替换 cosine 范式 (2026-03-30)

> Zen 5 解码器架构类比：整数/浮点/访存三执行单元共享前端，流水线重叠。

### 💡 核心灵感："cosine 不是天花板，是地基裂缝"

**问题链诊断**（来自 Stefan-CFD 多轮预取实验 0.2727 ≈ 0.2735）：
- Stefan 多轮预取本身有效（边界压力检测正确识别了扩展需求）
- 但探针用 cosine 检索全库 → 找回的文档还是 cosine 觉得好的 → 找不到 cosine 遗漏的文档
- **根因：整条管线的检索底座是 cosine，PDE 再怎么优化也突破不了 cosine 天花板**

**一凡的决断：把 cosine 从管线中彻底替换掉。**

### 🏗️ 完整管线设计

```
═══ 离线阶段（一次性）═══

全库文档 → Qwen3 推理 → 每文档 token-level hidden states (4096d)
         → 存入 Rust 引擎（每 doc = 一片 token 点云）
         → PQ 64×64 子空间切分表（AMX Tile 类比，已在 V8 验证）

═══ 在线阶段（每次查询）═══

Query → Qwen3 → query token 点云
    │
    ▼
Rust 内核: 全库 PQ-Chamfer 暴力检索
query 点云 × N doc 点云 → top-30
(SIMD 64×64 子空间并行, 替代 HNSW cosine)
    │
    ▼
Node.js: 建 KNN 图 + PDE → 浓度场
    │
    ▼
Stefan 多轮预取:
  探针 = 压力节点的 token 点云（不是单向量！）
  → Rust 内核: 探针点云 × 全库 PQ-Chamfer
  → 找到 cosine 永远找不到的文档
  → 扩池 → 重建图 → 重跑 PDE
  → 重复直到边界压力消失或池满
    │
    ▼
最终浓度排序 → 输出
```

### 🔧 Rust 内核架构：Zen 5 解码器类比

**一凡的类比洞察**：Zen 5 后端有整数 ALU、浮点 FPU、访存 Load/Store 三类执行单元，共享寄存器堆和缓存。Rust 内核应该复刻这个架构。

| Zen 5 执行单元 | Rust 内核对应 | 职责 |
|:--------------|:-------------|:-----|
| 整数 ALU (×4) | 图操作单元 | KNN 建图、边界检测、ID 管理、top-K 堆维护 |
| 浮点 FPU (×4, AVX-512) | 距离计算单元 | PQ-Chamfer SIMD 热循环（64×64 子空间内积） |
| 访存 Load/Store (×3) | 向量预取单元 | 从内存/磁盘加载 token 点云、cache line 预取 |

**共享资源（"公用的"）**：

| Zen 5 共享资源 | Rust 对应 |
|:--------------|:---------|
| 寄存器堆 / ROB | PQ 码本 (64×64 切分表) + 结果缓冲区 |
| L1/L2 Cache | 热文档点云缓存 (频繁访问的文档常驻) |
| 前端解码器 | 任务调度器 (决定下一批处理哪些文档) |

**三单元流水线重叠**：

```
时间 →
─────────────────────────────────────────────
访存:  [取 batch2 点云] [取 batch3 点云] [取 batch4 点云]
浮点:       [算 batch1 PQ距离] [算 batch2 PQ距离] [算 batch3 PQ距离]
整数:            [更新 batch0 top-K] [更新 batch1 top-K] [更新 batch2]
─────────────────────────────────────────────
         三条流水线重叠，吞吐量接近 3×
```

**Rust 实现路径**：`std::sync::mpsc` channel 连接三个线程池，`rayon` 做批内并行。

### ❌ V12 核心假设被证伪

**原假设**："cosine 是天花板，替换为 PQ-Chamfer 就能突破"
**实验结论**：**错误。** 当 query 只有 1 个向量（单点）时，PQ-Chamfer 不如 cosine 做初始检索。

**根因分析**：
- Chamfer distance 对**单点 vs 多点**退化为 "最佳句子匹配 + 所有句子平均惩罚"

- cosine 只看文档质心，天然对长文档友好
- **Chamfer 的优势在文档-文档（多点 vs 多点），不在 query-文档（单点 vs 多点）**

**修正结论**：
- cosine 粗筛 + Chamfer 精排 = 各取所长，这才是最优组合
- 要让 Chamfer 在初始检索中也优于 cosine，**必须先解决 P0 瓶颈：query 也必须是点云**
- Rust 内核架构正确，但需要等 V11（query token 点云化）才能发挥真正优势

## 十六、V13 灵感：PQ 子空间虚拟 Token — 不回模型的语义分解 (2026-03-30)

> 一凡原创直觉："核心依旧向量基础，用别的数学算法挖掘已有信息"

### 💡 核心灵感

**问题**：V11 要求回 Qwen3 提取 token hidden states（耗时、需 GPU）。能不能不回模型，从已有向量里提取更精细的信息？

**一凡的洞察**：4096 维向量的 64 个 PQ 子空间不只是距离计算的工具，它们是**独立的语义维度**。把每个子空间当作一个"虚拟 token"，单向量就变成了 64 点的点云。

```
行业标准: PQ 子空间 → 压缩加速（省内存）
一凡的用法: PQ 子空间 → 语义分解（增加匹配密度 ×64）
```

### 📐 数学描述

**vt_aligned（对齐虚拟 token Chamfer）**：

对每个子空间 s ∈ {0,...,63}，独立做句子级 Chamfer 距离，然后 64 个子空间取平均：

$$d_{vt}(q, D) = \frac{1}{64} \sum_{s=0}^{63} \left[ \min_{j} d_s(q, s_j) + \frac{1}{|D|} \sum_{j} d_s(s_j, q) \right]$$

其中 $d_s(a, b) = 1 - \frac{a[s] \cdot b[s]}{||a[s]|| \cdot ||b[s]||}$ 是第 s 个 64 维子空间的 cosine 距离。

**与 shape_pq64 的区别**：
- shape_pq64：先对 64 个子空间聚合成 1 个距离，再做句子级 Chamfer
- vt_aligned：先做句子级 Chamfer（在每个子空间内），再对 64 个子空间聚合
- **操作顺序不同 = 信息保留不同**

### ✅ 验证结果 (NFCorpus, 323 queries, 2026-03-30)

| Method | NDCG@10 | vs cosine | vs pq64 |
|:-------|:-:|:-:|:-:|
| cosine | 0.2195 | — | -19.5% |
| shape_pq64 | 0.2729 | +24.3% | baseline |
| vt_unaligned (跨子空间) | 0.2753 | +25.4% | +0.9% |
| v7_adjoint (伴随预取) | 0.2759 | +25.7% | +1.1% |

### 关键发现

1. **vt_aligned 打平历史最高**（0.2797 ≈ 0.2802），且不需要全库探针检索
2. **对齐 > 非对齐**（0.2797 > 0.2753）：跨子空间匹配引入噪声，与 V8 消融一致
3. **与 V7 正交**：vt_aligned 是距离函数改进，V7 是预取策略，可叠加
4. **零成本**：不需要模型推理，不需要新数据，纯数学重组已有句子向量

### 物理直觉

想象一个 4096 维向量是一栋建筑的蓝图：
- **cosine**：远远看一眼整体轮廓（1 个数字）
- **shape_pq64**：把蓝图拆成 64 张局部图纸，每张单独评分后取平均
- **vt_aligned**：对每张局部图纸，先在所有楼层（句子）里找最匹配的，再汇总

vt_aligned 比 pq64 好，因为"先在句子中找最优匹配"保留了更多局部信息。

## 2026-04-01：对流之死与图平滑之生

> 本节记录 2026-04-01 全天密集实验。核心结论：对流叙事在数据面前死亡，图拉普拉斯平滑取代 PDE 成为最优图处理方式，三数据集验证通过。

### 第一阶段：替代粗筛方案的全面验证与否定

#### 1. Token 倒排索引粗筛（V14）

**动机**：质心 cosine 粗筛本质是"单点 vs 单点"，与 Shape-CFD"一切皆点云"的哲学矛盾。目标是实现纯点云的端到端管线。

**实现**：Per-subspace K-means (64 子空间 × 256 中心)，对 880K tokens 建倒排表 56.4M 条目。三种查询变体——命中计数、距离加权 MaxSim、argmin 快速路径。

**结果**：

| 方法 | NDCG@10 | vs 质心 cosine |
|:--|:--:|:--:|
| 质心 cosine top-100 | 0.3220 | — |
| 倒排 top-200 p=1 | 0.3080 | -1.4% |
| 倒排 top-500 p=1 | 0.3175 | -0.45% |
| MaxSim 加权 | 0.0706 | 崩溃 |

**结论**：在 2473 文档规模上，质心作为全局代表的信噪比最高。倒排索引在小规模上无优势。MaxSim 加权根本性错误——centroid 距离不能代替 token 距离。倒排索引的价值要到 100K+ 才能体现。

**淘汰原因**：小规模下质心 cosine O(N) 扫描又快（23ms）又好，倒排索引引入了聚类近似误差但没换来速度优势。

#### 2. PQ 量化精排（ADC）

**动机**：880K tokens × 4096d × f32 = 13.65 GB 内存。PQ 量化可压缩 236 倍到 57.8 MB。

**实现**：复用倒排索引的 K-means 码本，每 token 编码为 64 字节 PQ code。ADC (Asymmetric Distance Computation) 查表替代 dot product。

**结果**：

| 方法 | NDCG@10 | 内存 |
|:--|:--:|:--:|
| 原始 f32 精排 | 0.3220 | 13.65 GB |
| PQ ADC 精排 | 0.2125 | 57.8 MB |

**结论**：PQ 256 码本精度不够。ADC 比 cosine 基线（0.2195）还差。压缩比 236 倍很好，但质量不可接受。

**淘汰原因**：256 个码本中心对 4096d 空间量化太粗。可能方案：增大码本到 1024+，或 ADC 只做粗筛不做精排。

#### 3. BM25 分级管线

**动机**：借鉴 Claude Code 的 Glob→Grep→Read 分级搜索，BM25 做第一层词汇粗筛。

**实现**：SQLite FTS5 全文索引，porter 词干化。

**结果**：

| 指标 | 值 |
|:--|:--:|
| BM25 Recall@200 | 28% |
| BM25 NDCG@10 | 0.3256（+48.3%，超过 cosine！）|
| BM25→cosine 精排 | 0.2648 |
| hybrid(BM25∪cosine)→Chamfer | 0.3209 |

**结论**：BM25 recall 太低（28%），数据处理不等式限制后续精排天花板。NFCorpus 生物医学术语变体严重（"heart attack" vs "myocardial infarction"）。BM25 单独做得好是 NFCorpus 的特殊性，不是通用方案。

**淘汰原因**：recall 天花板。但 BM25 ∪ cosine 混合召回 + Chamfer 精排 = 0.3209，接近最优，说明多路召回有价值。

### 第二阶段：对流之死

#### 4. 图拉普拉斯平滑 vs PDE

**动机**：数学审查指出 Pe≈0.064（极度扩散主导），PDE 可能退化为图拉普拉斯平滑。需要实验验证。

**实现**：`laplacian_smooth(c0, adj, alpha=0.02, steps=20)`——纯扩散，无对流，无反应项。与 `solve_cfd`（完整 PDE）在相同图结构和 C₀ 上对比。

**结果（NFCorpus 323 queries）**：

| 方法 | NDCG@10 |
|:--|:--:|
| 图拉普拉斯平滑 | **0.2900** |
| Shape-CFD V10（PDE） | 0.2852 |

**宣判**：图平滑 > PDE。对流项不但没用，还是负贡献（-0.5 个百分点）。

#### 5. Allen-Cahn 反应项验证

**动机**：高模 Opus 建议对流可能不行，但 Allen-Cahn 非线性项（三次项）做不到的事图平滑做不到——极化。

**结果**：

| 方法 | NDCG@10 |
|:--|:--:|
| 纯图平滑 | **0.2900** |
| 图平滑 + AC γ=0.2 | 0.2881 |
| 图平滑 + AC γ=0.5 | 0.2875 |
| 图平滑 + AC γ=2.0 | 0.2869 |

**宣判**：Allen-Cahn 也是负贡献。gamma 越大越差。极化过程损伤了图平滑的去噪效果。

#### 6. Token 梯度对流

**动机**：旧对流失败是因为质心差方向随机。新思路：用 token Chamfer 分数梯度驱动对流，实现跨粒度能量传递。

**数学审查**：守恒性通过。Kendall tau = 0.378 证明 token 和 sentence 排名高度互补。C₀ 用句子级（方案 A）= 真正跨粒度融合。

**结果**：

| 方法 | NDCG@10 |
|:--|:--:|
| 旧 PDE | 0.2852 |
| Token 梯度 β=1 | 0.2775 |
| Token 梯度 β=3 | 0.2618 |
| Token 梯度 β=5 | 0.2533 |

**宣判**：β 越大越差。"富者愈富"效应——对流把浓度从低分文档抽走灌给高分文档，抑制了 KNN 图的邻域修正信号。

#### 对流死因总结

**根因**：不是对流公式错了，是 regime 不对。
- 候选集 55 文档经 cosine 粗筛后语义太近，|c_j - c_i| 方向随机
- Pe ≈ 0.064，对流被扩散淹没
- 更强的对流（token 梯度）反而有害——强化已有排名而非修正它
- 在精排候选集上，纯扩散（邻域平均）是最优信息提取方式

**可能复活场景**：更大候选集、粗筛阶段、更异构的数据分布。一凡有"对流在粗筛复活"的新想法，待后续验证。

### 第三阶段：统计显著性与融合探索

#### 7. Bootstrap 显著性检验

**结果（323 queries, 10000 bootstrap）**：

| 对比 | Diff | p-value | 显著？ |
|:--|:--:|:--:|:--:|
| fusion_07 vs token_2stage | +0.0012 | 0.3346 | **不显著** |
| token_2stage vs cosine | +0.1024 | <0.001 | *** |
| shape_cfd vs cosine | +0.0656 | <0.001 | *** |

**宣判**：PDE 融合增量 +0.0012 是统计噪声（p=0.33，95% CI 跨零）。token_2stage +46.7% 才是真正的核心贡献。

#### 8. Helmholtz-Kalman 融合

**动机**：Helmholtz 分解提取图信号中与 token 正交的结构性修正，低通滤波去噪。Kalman 增益实现 per-document 自适应权重。

**数学框架**：
- 正交残差：r = s_G - α*s_T
- 低通滤波：(I + β·L_norm) · r_smooth = r
- Token 不确定度：σ²_T = Var(per-token MaxSim)
- 图不确定度：σ²_G = 邻域分数不一致性
- 融合：s_fused = s_T + (1 - K_i) · r_smooth，K_i = σ²_G / (σ²_T + σ²_G)

**结果**：
- 第一版（cosine 粗筛候选集）：0.3036（差——候选集质量拖后腿）
- 修复后（token 粗筛候选集）：0.3231（β=0.5），追平旧 fusion 0.3232

**宣判**：HK 融合没有超越简单 0.7/0.3 线性融合。Per-document 自适应在 NFCorpus 上没有比全局权重更好。瓶颈不在融合方式，在图信号本身的增量有限。

#### 11. 新增证伪（2026-04-02）

**密度加权 Chamfer（证伪 #18）**：
- 假说：密集区域的点更重要，用密度加权 Chamfer 应该更好
- 实验：密集区是冗余不是重要性，均匀加权最优
- 结果：NDCG -0.9% ~ -1.3%
- 结论：密集区反映的是嵌入模型的冗余分布，不是语义重要性

**PQ 码重建向量（证伪 #19）**：
- 假说：用 PQ 码重建原始向量再计算 Chamfer
- 实验：重建向量质量极差，Chamfer min 操作放大量化误差
- 结果：NDCG 暴跌 34%
- 结论：PQ 码适合做子空间独立距离，不适合重建完整向量
